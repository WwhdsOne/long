# Boss 房间部位显示与掉血同步 Bug 排查记录

## 背景

Boss 房间存在一组连锁问题：

1. 首次进入房间时 Boss 部位完整，后续刷新后部分部位被覆盖消失。
2. 修复部位消失后，部位血量又出现不同步。
3. Boss 被击败切到下一只 Boss 时，新 Boss 首帧偶发缺少部位，刷新后恢复。
4. 第一行 / 第一列，尤其 `x=0` 或 `y=0` 的格子，更容易出现“部位不显示掉血”。

这些现象都出现在前端实时状态合并链路，不是后端持久化数据真的缺失。

## 根因拆分

### 1. 增量 Boss 被当前 Boss 整对象覆盖

位置：

- `frontend/src/utils/bossState.js`

旧逻辑把同一只活动 Boss 的后续 `incoming boss` 直接当完整快照覆盖。

问题：

- 当后续消息没带完整 `parts`，或只带部分 `parts` 时，会把首屏完整部位清空或截断。

修复：

- 同一只活动 Boss 改为按坐标合并 `parts`。
- 缺失部位保留旧值，不再整数组替换。

### 2. click_ack 的部位 delta 没有真正落到前端状态

位置：

- `frontend/src/pages/publicPageState.js`
- `frontend/src/utils/bossState.js`

后端 `click_ack` 设计上只发：

- `partStateDeltas`
- 伤害
- 个人资源增量

不发完整 `boss`。

旧前端只拿 `partStateDeltas` 做特效和跳字，没有更新 `boss.parts.currentHp`。

问题：

- 服务端部位已经掉血，但页面局部血条不动。

修复：

- 新增 `applyBossPartStateDeltas`。
- 收到 `click_ack` 后按 `x/y` 把 `afterHp` 写回对应部位。
- 同步更新 Boss 总血量。
- 过时 delta 不允许把血量回抬。

### 3. Boss 切换窗口里，旧 Boss 的晚到 delta 污染新 Boss

位置：

- `frontend/src/utils/bossState.js`

问题：

- 上一只 Boss 被击杀后，最后一条 `partStateDeltas` 可能晚于下一只 Boss 的 `public_delta` 到达。
- 前端之前只看坐标，不看时序，就把旧 Boss 的 `afterHp=0` 套到新 Boss 同坐标部位上。
- 结果是新 Boss 首帧看起来“缺部位”或“部位已死”。

修复：

- 应用 delta 时加入 `beforeHp` 校验。
- 只有当前部位血量不高于该 delta 的 `beforeHp`，才允许应用。

### 4. 公共 boss 更新的总血量防回退，误吞了部位掉血

位置：

- `frontend/src/utils/bossState.js`

旧逻辑：

- 如果 `incoming.currentHp > current.currentHp`，直接丢弃整包 `incoming boss`。

问题：

- 某些乱序公共包里，Boss 总血量虽然看起来更旧，但 `parts.currentHp` 其实已经更新。
- 整包丢弃会把有效的部位掉血一起吞掉。

修复：

- Boss 总血量继续保留“只向下更新”。
- 但 `parts` 仍按坐标继续合并。
- 每个部位的 `currentHp` 也只允许向下变化。

### 5. protobuf 解码把 `0` 坐标吃掉

位置：

- `frontend/src/utils/realtimeProto.js`

问题：

- WebSocket 二进制消息经过 protobuf `toObject()` 时，proto3 默认值 `0` 可能被省略。
- 这样 `x=0` 或 `y=0` 的 `BossPart` / `BossPartStateDelta` 到前端后会丢失坐标字段。
- 直接命中第一行、第一列，所以你观察到 `0 行 0 列似乎都有问题`。

修复：

- 外层消息仍保持稀疏解码。
- 但嵌套的 `BossPart` 和 `BossPartStateDelta` 单独用带默认值的解码参数转换。
- 保证 `x: 0`、`y: 0` 不会被省略。

## 实际影响链路

### 首次进入 Boss 房间

- HTTP `/api/battle/state` 或 SSE 首包是完整快照。
- 所以最开始经常看起来是正常的。

### 后续实时更新

- WebSocket `public_delta` 和 `click_ack` 进入前端后，会经过多层合并。
- 任一层把 `parts`、`partStateDeltas`、`x/y=0`、或部位血量吞掉，就会出现：
  - 部位消失
  - 部位不掉血
  - 新 Boss 首帧残缺

## 已补回归测试

- `src/utils/bossState.test.js`
  - 缺失 `parts` 时保留旧部位
  - 只带部分 `parts` 时按坐标合并
  - `click_ack` 部位 delta 同步部位血量和 Boss 总血
  - 旧 Boss 晚到 delta 不污染新 Boss
  - 总血量乱序时，部位掉血仍可更新
- `src/utils/realtimeTransport.test.js`
  - `click_ack` 的 `x=0 / y=0` delta 解码不丢坐标
  - `public_delta` 的 `x=0 / y=0` 部位解码不丢坐标

## 结论

这不是单点 bug，而是一组“实时增量状态合并 + protobuf 默认值 + Boss 切换时序”叠加出来的问题。

后端 Boss 数据主体基本是正确的，核心故障集中在前端：

1. 把增量当全量。
2. 没消费部位 delta。
3. 用总血量回退保护误伤部位血量。
4. 没处理 Boss 切换窗口里的晚到消息。
5. protobuf 把 `0` 坐标省略，导致第一行 / 第一列异常最明显。
