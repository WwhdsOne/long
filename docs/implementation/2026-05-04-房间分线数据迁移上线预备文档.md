# 房间分线数据迁移上线预备文档

## 目标

为本次房间分线上线准备一套可执行的数据迁移与上线检查清单，避免只完成代码部署而遗漏 Redis / Mongo 的历史数据、回填数据和上线后校验。

本文基于：

- [docs/architecture/2026-05-03-房间分线方案.md](../architecture/2026-05-03-房间分线方案.md)
- 当前仓库已落地的迁移工具：
  - `backend/cmd/migrateroommodel`
  - `backend/cmd/backfillbosskills`
  - `backend/cmd/checkbosskills`

## 本次上线涉及的数据面

### Redis

- `boss:current -> boss:current:1`
- `boss:cycle -> boss:cycle:1`
- `player:room:{nickname}`
- `player:room:cd:{nickname}`
- `resource:{nickname}.boss_kills`
- `total:boss:kills`

说明：

- `boss:current` 和 `boss:cycle` 是本次必须迁移的旧全局 key。
- `player:room:*` 与 `player:room:cd:*` 允许惰性初始化，不要求上线前全量预写。
- `boss_kills` 和 `total:boss:kills` 不属于房间模型迁移本体，但属于本次页面改动依赖的展示数据，建议在正式切流前一起回填。

### Mongo

本次必须检查并回填三张集合：

- `boss_history`
- `domain_events`
- `admin_audit_logs`

回填目标：

- `room_id="1"`
- `queue_id="1"`

并确认对应索引已创建。

## 上线前检查

### 1. 配置检查

确认运行配置中：

- `mongo.enabled=true`
- `room.enabled=true`
- `room.default_room="1"`
- `room.ids` 已包含计划开放的房间

如果 `mongo.enabled=false`，`migrateroommodel` 无法执行。

### 2. 代码与工具检查

至少确认以下命令可执行：

```bash
go -C backend run ./cmd/migrateroommodel plan
go -C backend run ./cmd/migrateroommodel verify
go -C backend run ./cmd/checkbosskills
```

说明：

- `migrateroommodel` 当前没有挂到 `Makefile`，需要直接用 `go -C backend run ./cmd/migrateroommodel ...`
- `backfillbosskills` 与 `checkbosskills` 已有 `Makefile` 入口，也可直接运行 Go 命令

### 3. 迁移前数据探查

先执行：

```bash
go -C backend run ./cmd/migrateroommodel plan
```

期望输出至少包含：

- 旧 Redis `boss:current` 是否存在
- 旧 Redis `boss:cycle` 是否存在
- Mongo 三张集合还有多少历史文档未补 `room_id` / `queue_id`

如果这里已经发现目标 key 不存在、或 Mongo 已全部回填，要先判断是已迁过，还是环境指错，不能直接继续。

### 4. Boss 击杀展示数据检查

先执行：

```bash
go -C backend run ./cmd/checkbosskills
```

重点确认：

- `TotalBossKillsKey` 是否可读
- `TotalBossKills` 是否非空
- `PlayersWithBossKills` 是否已有数据

如果 `PlayersWithBossKills: <none>`，不代表房间迁移失败，但说明前端“我的Boss击杀数 / 总Boss击杀数”很可能仍未准备好，需要在部署后补跑击杀回填。

## 备份范围

停机前至少备份以下数据。

### Redis 关键键

- `{ns}boss:current`
- `{ns}boss:cycle`
- `{ns}boss:current:1`
- `{ns}boss:cycle:1`
- `{ns}player:room:*`
- `{ns}player:room:cd:*`
- `{ns}resource:*`
- `{ns}total:boss:kills`
- `{ns}boss:*:damage`

建议至少保留：

- 旧全局 Boss / 循环 key 快照
- 全量 `resource:*` 快照
- 当前环境 `RDB` 或等价全量备份

### Mongo 集合

- `boss_history`
- `domain_events`
- `admin_audit_logs`

要求：

- 至少留一份迁移前导出
- 能按集合单独恢复

## 正式上线步骤

### 步骤 1：停机

停服务，阻断以下写入：

- 前台点击
- 离页挂机结算
- 后台 Boss 操作
- 后台循环开关与跳过操作

不要在服务仍对外提供写能力时执行迁移。

### 步骤 2：执行房间模型迁移

先看计划：

```bash
go -C backend run ./cmd/migrateroommodel plan
```

正式迁移：

```bash
go -C backend run ./cmd/migrateroommodel migrate
```

迁移内容包括：

- `boss:current -> boss:current:1`
- `boss:cycle -> boss:cycle:1`
- 给 `boss:current:1` 补 `room_id=1`、`queue_id=1`
- 给 `boss:cycle:1` 补 `queue_id=1`
- Mongo 三张集合补 `room_id=1`、`queue_id=1`
- 创建新增 Mongo 索引

### 步骤 3：执行房间模型校验

```bash
go -C backend run ./cmd/migrateroommodel verify
```

校验通过的最低标准：

- 旧 `boss:current` 不再存在
- 旧 `boss:cycle` 不再存在
- `boss:current:1` 的 `room_id` 合法
- 三张 Mongo 集合不存在缺失 `room_id` / `queue_id` 的旧文档

### 步骤 4：部署新版本

部署包含以下能力的新版本：

- `hall` 大厅态与房间态拆分
- 房间独立 Boss / 循环模型
- 房间切换与退出规则
- 按房间读取 Boss 状态与实时推送
- Boss 击杀数读取与历史回退逻辑

### 步骤 5：启动服务后补 Boss 击杀统计

建议在新版本启动后立即执行：

```bash
make backend-backfill-boss-kills
```

等价命令：

```bash
go -C backend run ./cmd/backfillbosskills
```

该回填会：

- 统计每个玩家参与过多少个历史 Boss
- 优先读取 Redis `boss:{bossID}:damage`
- Redis 不存在时回退到 Mongo `boss_history.damage`
- 写回 `resource:{nickname}` 的 `boss_kills`
- 写回全局 `total:boss:kills`

### 步骤 6：检查 Boss 击杀统计

```bash
make backend-check-boss-kills
```

等价命令：

```bash
go -C backend run ./cmd/checkbosskills
```

如果要抽查单人：

```bash
go -C backend run ./cmd/checkbosskills --nickname 某个玩家昵称
```

注意：

- 检查工具当前读的是 `resource:{nickname}.boss_kills`
- 不是旧的 `user:{nickname}` 体系

## 上线后业务验收

### 房间模型验收

至少验证：

1. 房间 `1` 继承旧全局当前 Boss。
2. 房间 `1` 继承旧全局循环状态和循环队列。
3. 不同房间的 `boss.id`、`currentHp`、Boss 伤害榜相互独立。
4. 大厅只显示房间选择与点击总榜，不显示 Boss 战斗区。
5. 战斗房间只显示 Boss 战斗区与 Boss 榜，不显示大厅分流区。
6. 退出到大厅逻辑可用；重新加入战斗房间时遵守冷却规则。

### Boss 击杀展示验收

至少验证：

1. `总Boss击杀数` 能读到 `total:boss:kills`
2. `我的Boss击杀数` 能读到 `resource:{nickname}.boss_kills`
3. 对于只有 Mongo 历史、没有 Redis 伤害榜的旧 Boss，回填后也能统计到参与次数

### Mongo 验收

抽样验证三张集合新老数据都带：

- `room_id`
- `queue_id`

并确认后台历史与日志检索未因新增字段失效。

## 回滚约束

本次回滚只支持：

- 代码版本回退
- 配置关闭 `room.enabled`

不支持：

- 把分线启用后的多房间进度无损压回旧单房间模型

原因：

- 旧版只理解 `boss:current` / `boss:cycle`
- 新版启用后，其他房间可能已经产生独立 Boss 进度

如果必须回滚到旧单房间模型且保留房间 `1` 进度，需要额外执行反向迁移，把：

- `boss:current:1 -> boss:current`
- `boss:cycle:1 -> boss:cycle`

这不属于本次预备文档默认步骤。

## 建议执行顺序

建议按下面顺序执行，避免漏项：

1. `go -C backend run ./cmd/migrateroommodel plan`
2. 停机
3. 备份 Redis / Mongo
4. `go -C backend run ./cmd/migrateroommodel migrate`
5. `go -C backend run ./cmd/migrateroommodel verify`
6. 部署并启动新版本
7. `make backend-backfill-boss-kills`
8. `make backend-check-boss-kills`
9. 前台和后台按房间能力做业务验收

## 上线负责人备注

- 如果当前环境已经跑过一次 `migrateroommodel migrate`，再次执行前要先确认 `boss:current:1` / `boss:cycle:1` 是否已存在，避免命令因“目标 key 已存在，拒绝覆盖”中断。
- 如果当前环境的 `boss_kills` 已经存在，重复跑 `backfillbosskills` 是允许的；它是重建式写回，不依赖旧值累加。
- 如果 `checkbosskills` 显示总击杀有值、个人击杀为空，优先检查 Mongo `boss_history.damage` 或 Redis `boss:{bossID}:damage` 是否有有效参与数据，而不是先怀疑前端展示。
