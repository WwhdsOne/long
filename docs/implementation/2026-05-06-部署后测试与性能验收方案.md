# 部署后测试与性能验收方案

## 目标

本文现在只保留一个用途：

- 对“单一点击链路”做部署后专项测试
- 对比 3 个阶段的数据：
  - 当前
  - 优化后
  - `Snappy + sync.Pool` 再次优化后

不再覆盖：

- 后台接口
- 装备链路
- 任务链路
- SSE 持续连接
- 多连接并发压测
- 其它烟雾回归

## 测试边界

本方案只测这一段：

- 玩家登录
- 单个 `WebSocket /api/ws`
- 单次二进制 `click`
- 收到对应的 `click_ack`

更具体地说：

- 只用 1 条 WebSocket 连接
- 顺序发点击
- 每次都等待确认后再发下一次
- 关心的是“点击发出到 `click_ack` 收到”的时间

这套方案测的是：

- 单连接顺序点击时，平均能确认多少次点击/秒
- 单连接顺序点击时，`p50 / p95 / p99` 延迟是多少

它**不是**多连接极限压测，也不是全站容量压测。

## 前提约束

为保证 3 轮数据能横向对比，下面这些条件必须尽量保持一致：

- 同一个测试账号
- 同一个站点域名
- 同一个房间
- 同一个 Boss
- 同一个部位坐标
- 同一个点击次数
- 同一台压测机
- 尽量相近的网络环境
- 尽量避开 Boss 快死、频繁切房、在线人数剧烈波动时段

否则“当前 / 优化后 / `Snappy + sync.Pool` 再次优化后”之间的结果不能直接比较。

## 当前脚本口径

本方案默认使用本地脚本：

- [local_click_latency 脚本使用说明](./../developer-reference/2026-05-06-local-click-latency-脚本使用说明.md)

脚本当前口径是：

- 登录指定玩家账号
- 建立 1 条 `WebSocket`
- 顺序发送二进制点击
- 忽略异步推送（例如 `online_count`）
- 只在收到真正的 `click_ack` 时计入一次点击完成

## 关键事实

### 1. `slug` 不代表房间

当前后端只接受 `boss-part:x-y` 形式的点击目标。

也就是说：

- `-slug boss-part:0-0` 表示“点击当前房间 Boss 的 `(0,0)` 部位”
- 不是“点击 0 号房间”
- 不是“自动找一个可点的目标”

### 2. 房间来自玩家当前房间

脚本不会自己切房间。

真正生效的是：

- 这个账号当前保存在哪个房间

所以每次正式测试前都要先确认：

- 测试账号已经切到目标房间
- 目标房间当前有活跃 Boss
- 你要测的 `boss-part:x-y` 确实存在而且可攻击

## 标准测试流程

### 1. 固定测试参数

建议先固定下面这些值：

```bash
export BASE_URL='https://www.wclick.top'
export TEST_NICKNAME='233'
export TEST_PASSWORD='233'
export TEST_ROOM_ID='2'
export TEST_SLUG='boss-part:0-0'
export TEST_COUNT='500'
```

如果你已经确定了别的房间和部位，就统一替换成你实际要测的值。

### 2. 先切到目标房间

推荐直接在前台页面里，用这个账号手动切房间。

如果要用接口，可以这样做：

```bash
curl -c /tmp/long.cookies -b /tmp/long.cookies \
  -H 'Content-Type: application/json' \
  -d "{\"nickname\":\"$TEST_NICKNAME\",\"password\":\"$TEST_PASSWORD\"}" \
  "$BASE_URL/api/player/auth/login"

curl -c /tmp/long.cookies -b /tmp/long.cookies \
  -H 'Content-Type: application/json' \
  -d "{\"roomId\":\"$TEST_ROOM_ID\"}" \
  "$BASE_URL/api/rooms/join"
```

### 3. 确认目标部位

确保：

- 当前房间 Boss 还活着
- `TEST_SLUG` 对应的部位还活着
- 不要在 Boss 临死前做正式记录

### 4. 预热一次

先跑一轮小样本，确认链路通畅：

```bash
go -C backend run ./cmd/local_click_latency \
  -base "$BASE_URL" \
  -nickname "$TEST_NICKNAME" \
  -password "$TEST_PASSWORD" \
  -slug "$TEST_SLUG" \
  -count 20
```

只要没有这些错误，就可以继续：

- `UNAUTHORIZED`
- `TOO_MANY_REQUESTS`
- `BOSS_PART_NOT_FOUND`
- `Boss 部位已被击碎`

### 5. 正式记录

正式记录统一用同一条命令：

```bash
go -C backend run ./cmd/local_click_latency \
  -base "$BASE_URL" \
  -nickname "$TEST_NICKNAME" \
  -password "$TEST_PASSWORD" \
  -slug "$TEST_SLUG" \
  -count "$TEST_COUNT"
```

建议每个阶段至少跑 3 次，然后把结果手工填进记录表。

## 每轮需要记录什么

脚本输出里建议至少记录这些字段：

- `样本数`
- `总耗时`
- `平均吞吐`
- `最小延迟`
- `平均延迟`
- `p50`
- `p95`
- `p99`
- `最大延迟`
- 是否出现业务错误

如果你想把环境信息也一起保留，建议再补：

- 提交号 / 镜像版本
- 房间 ID
- 部位 slug
- 测试时间
- 是否开白名单
- 是否开 `Snappy`
- 是否开 `sync.Pool`

## 通过标准

这份文档不再写死统一阈值，因为你这次的核心是“前后版本对比”，不是“是否满足一个绝对 SLA”。

这轮真正要看的是：

- 优化后相对当前，`平均吞吐` 是否上升
- 优化后相对当前，`平均延迟 / p95 / p99` 是否下降
- `Snappy + sync.Pool` 再次优化后，是否继续改善
- 是否引入新的错误、抖动或长尾恶化

如果出现下面任一情况，这轮数据就不应直接纳入对比：

- Boss 在测试途中死亡
- 部位在测试途中死亡
- 切房了
- 账号被踢下线
- 实际打到了不同房间或不同部位

## 推荐使用方式

推荐的对比顺序：

1. 当前
2. 优化后
3. `Snappy + sync.Pool` 再次优化后

每个阶段都：

1. 切到同一房间
2. 确认同一部位坐标
3. 跑相同的 `count`
4. 把结果填到同一张表

这样最后你看到的差异才有参考价值。
