# WebSocket + Protobuf 点击链路优化方案

> 状态：已审核，待实现
> 日期：2026-05-04
> 结论：保留现有 `WebSocket` 实时主链路，保留“一次点击一条消息”的交互模型，不引入 `gRPC`，不引入 `Envoy`；高频优化集中在 `protobuf over websocket`、消息瘦身、下行增量化和兼容切换。

## 一、结论先行

当前点击链路的优化方向正式收敛为：

- 保留浏览器到服务端的 `WebSocket`
- 保留“一次点击发送一条消息”
- 不做 `50ms / 100ms` 批量聚合
- 不引入 `gRPC`
- 不引入 `Envoy`
- 引入 `protobuf`，用于压缩高频消息体和降低编解码成本

这个结论的核心前提只有一个：

- 当前场景最重要的是 **手感优先**

只要客户端把多次点击合并成一包，就会引入明显粘滞感。既然这一点已经被真实体验验证过，就不应该再为了理论吞吐去牺牲输入反馈。

## 二、为什么保留 WebSocket

### 2.1 当前问题不是“需要换成 gRPC”

浏览器实时双向通信里，`WebSocket` 本身就是最自然的选择。

当前高频压力主要来自：

- 点击消息发送频率高
- 下行战斗同步频率高
- 当前协议是文本 JSON，消息体和编解码开销偏大

这些问题并不要求换掉 `WebSocket` 连接模型。

### 2.2 gRPC 不适合当前目标

如果要让浏览器直接走 gRPC，现实上通常是：

- `gRPC-Web`
- 或额外网关做协议桥接

这样会带来：

- 接入复杂度增加
- 浏览器调试链路更重
- 实时双向交互语义并不比 `WebSocket` 更直接

在“高频点击 + 高频增量同步”的场景里，`gRPC` 不是必要条件。

### 2.3 批量聚合不适合当前手感目标

此前已经试过 `50ms / 100ms` 合帧，结论是：

- 吞吐可能更漂亮
- 但玩家手感明显发黏

因此本方案明确拒绝：

- 客户端点击批量上报
- 人为延迟点击确认
- 以聚合换取表面吞吐

## 三、优化目标

本方案要解决的不是“减少点击条数”，而是“让每条高频消息更轻、更快、更少浪费”。

首版目标：

- 保持一次点击一条消息
- 保持点击反馈手感不变
- 压缩高频消息体
- 降低前后端 JSON 编解码开销
- 进一步收紧下行同步载荷
- 为后续压测和观测建立稳定协议基线

不做目标：

- 不做浏览器端 gRPC 化
- 不做接入层网关升级
- 不做客户端定时批量聚合
- 不做一次性全协议翻新

## 四、协议收敛方案

### 4.1 总体策略

采用：

- `protobuf over websocket`

而不是：

- `gRPC over websocket`
- `gRPC-Web`
- 自定义无 schema 二进制协议

原因：

- `protobuf` 已经能解决消息体和编解码问题
- 保留现有 `WebSocket` 会话模型，改造面更可控
- 有 schema，便于版本管理和前后端联调

### 4.2 首批改造消息

优先只改最热路径。

第一批：

- 上行 `click`
- 下行 `click_ack`
- 下行 Boss 部位 HP / 伤害增量 / 天赋触发事件

第二批：

- 高频个人态增量
- 高频公共态增量

最后再看：

- `snapshot`
- 低频管理接口

原则：

- 高频先改
- 低频后改
- 不要求一口气把所有实时消息都二进制化

### 4.3 保持“一次点击一条消息”

点击消息模型保持不变：

- 用户点击一次
- 立即发一条上行消息
- 服务端立即处理
- 立即回 `ack` 或增量状态

这条约束是本方案的硬边界，不允许为了吞吐把上行点击改成时间窗口聚合。

## 五、实现策略

### 5.1 双协议兼容期

首版必须允许：

- 旧客户端继续发 JSON `click`
- 新客户端改发 protobuf `click`

服务端短期同时支持两套解析路径：

- JSON 文本帧
- protobuf 二进制帧

这样做的目的：

- 避免前后端必须同一时间切换
- 降低上线风险
- 允许灰度验证二进制协议链路

兼容期结束后，再删除旧 JSON 高频路径。

### 5.2 高频消息单独 schema

不要把整套实时协议一次性塞进一个超大 proto。

建议拆成独立高频消息定义：

- `ClickRequest`
- `ClickAck`
- `BattleDelta`
- `UserDelta`
- `TalentTriggerEvent`

要求：

- 字段最小化
- 可选字段只保留高频必需项
- 避免把大块冗余状态重复塞进每次增量

### 5.3 下行同步继续瘦身

引入 protobuf 以后，仍然不能放任下行消息膨胀。

要继续坚持：

- 高频只发增量
- 非必要字段不重复发送
- 能通过本地状态机推导的，不重复广播
- 大对象只在初始化或显式刷新时下发

否则只是把“大 JSON”变成“大 protobuf”，收益会被浪费掉。

## 六、服务端改造边界

首版服务端改造应收敛在实时层，不扩散到核心战斗逻辑。

建议边界：

- `httpapi/realtime_socket.go`
  - 识别文本帧 / 二进制帧
  - 路由到 JSON 或 protobuf 解码路径
- 高频消息编解码模块
  - 集中处理 protobuf schema 与转换
- 业务执行层
  - 继续复用现有 `executeButtonClick`
  - 不重写点击业务逻辑
- 事件回写层
  - 高频 `ack / delta` 支持 protobuf 序列化

明确不做：

- 不重写 `Store.ClickButton`
- 不把战斗逻辑改造成 protobuf 驱动
- 不顺手做无关协议重构

## 七、前端改造边界

前端首版改造目标也应最小化：

- 保留现有 `BattlePage` 点击交互模型
- 保留现有 WebSocket 连接管理
- 新增 protobuf 编解码能力
- 只替换高频消息收发格式

不做：

- 不改用户点击节流策略
- 不改页面交互节奏
- 不强制全站所有页面同时切换

## 八、验收标准

协议切换完成后，至少要验证：

- 一次点击一条消息，手感与当前一致
- 高频点击场景下，客户端没有明显粘滞
- 上下行平均消息体明显缩小
- 前后端高频编解码 CPU 下降
- 点击确认与战斗增量没有出现乱序或丢失
- JSON 兼容客户端与 protobuf 客户端可并存一段时间

压测和观测建议重点看：

- WebSocket 入站消息 QPS
- WebSocket 出站消息 QPS
- 平均消息体大小
- p50 / p95 / p99 点击确认延迟
- 服务端 CPU
- 前端主线程压力

## 九、最终建议

当前点击链路的正式优化路线应改写为：

- `WebSocket` 保留
- `gRPC` 放弃
- `Envoy` 放弃
- 批量点击放弃
- `protobuf` 引入
- 高频消息优先改造
- 下行增量继续瘦身

如果后续仍然有性能瓶颈，再继续往下看：

- 更激进的二进制消息裁剪
- 更细的事件分层
- 更强的前端本地状态复用

但在这一步之前，没有必要为了“协议名字更高级”去上 `gRPC`。
