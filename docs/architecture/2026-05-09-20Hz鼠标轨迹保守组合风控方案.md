# 20Hz 鼠标轨迹保守组合风控方案

> 状态：待评审
> 日期：2026-05-09
> 结论：采用“20Hz 轻采样 + 点击前摘要上传 + 后端低权重组合记分”的保守方案；单独“无轨迹”不计分，不把鼠标轨迹作为硬封禁依据。

## 一、背景

当前项目已经有统一账号积分制 `anti_script`：

- 点击过快记分
- 登录 Turnstile 明确失败记分
- 购买体力 Turnstile 明确失败记分
- 购买体力后立即手点记分

但这套规则主要还是围绕“频率”和“明确失败事件”。

对于以下脚本，现有规则仍然缺少额外信号：

- Playwright / Puppeteer 直接 `click()`，没有点击前鼠标移动轨迹
- 补了 `mouse.move()` 的脚本，但轨迹过于平滑、重复、程序化
- 压着频率阈值运行的浏览器自动化脚本

同时，项目也确认了几个不能误伤的真实场景：

- 触摸板用户可能基本不移动光标，只在一个点持续点击
- 鼠标用户也可能长时间停在同一个 Boss 部位猛点
- 校园网 / 公司网共用 IP，不能继续把“同 IP”当强信号

因此，本方案的目标不是“用轨迹直接判脚本”，而是：

> 在不拖慢高频点击交互的前提下，为现有 `anti_script` 增加一个低误伤、低权重、可组合的鼠标轨迹信号。

## 二、目标与非目标

### 2.1 目标

- 前端以低开销方式采集最近短窗口鼠标轨迹。
- 点击时只上传轨迹摘要，不上传完整长轨迹。
- 后端在点击热路径上只做 O(1) 轻量规则判断。
- 轨迹信号只作为组合风控的一部分，不单独构成处罚结论。
- 不影响“定点持续点击”的正常玩家体验。

### 2.2 非目标

- 不尝试识别真实“系统级设备码”。
- 不尝试 100% 识别隔离浏览器、多 profile 或高级自动化。
- 不引入复杂模型、AI 分类器或离线训练链路。
- 不把完整轨迹长期保存到 Redis / Mongo。
- 不把“无轨迹”直接等同于脚本。

## 三、核心口径

### 3.1 总原则

轨迹只做“弱风控信号”：

- 单次命中不应直接封禁
- 单独无轨迹不记分
- 只有“程序化轨迹”与其他异常信号叠加时，才提高风险分

### 3.2 保守判定口径

首版只保留三条保守规则：

1. 点击前存在轨迹，但轨迹明显程序化
2. 程序化轨迹在短时间内连续重复出现
3. 程序化轨迹与现有高频点击异常同时出现

明确不做的规则：

- 点击前没有 `mousemove` 就记分
- 长时间停在一个点点击就记分
- 轨迹点数少就直接记分

### 3.3 为什么单独“无轨迹”不记分

这是本方案最重要的误伤控制点。

以下用户都可能天然“轨迹很少甚至没有”：

- 触摸板用户
- 已将光标停在弱点位置持续点击的鼠标用户
- 窗口焦点切换后回到页面直接点击的用户

因此：

- `no movement before click` 只能作为上下文信息
- 不能单独进入 `anti_script` 积分

## 四、前端采样方案

### 4.1 采样频率

固定采用 `20Hz`。

也就是：

- 最多每 `50ms` 记录一个采样点
- 浏览器原始 `mousemove` 事件频率更高时，主动降采样

原因：

- 比逐事件采集轻得多
- 对“点击前 1~2 秒轨迹特征”已经足够
- 在高频点击页面里对主线程压力可控

### 4.2 采样窗口

只保留最近 `1500ms` 到 `2000ms` 的轨迹。

推荐首版窗口：

- `1500ms`

采用前端内存环形缓冲区，不做长期累计。

### 4.3 单点结构

前端缓冲区里每个采样点只保留：

- `x`
- `y`
- `t`

其中：

- `x` / `y` 为相对视口坐标
- `t` 为 `performance.now()` 或等价高精度时间戳

不额外附带：

- DOM 路径
- 页面全量状态
- 原始浏览器事件对象

### 4.4 点击时只提取摘要

点击发生时，不上传原始点数组，只从缓冲区现算一个小摘要。

摘要建议包含：

- `sampleCount`
- `windowMs`
- `distanceTotal`
- `distanceStraight`
- `detourRatio`
- `speedAvg`
- `speedStdLike`
- `directionChanges`
- `pauseCount`
- `hasEffectiveMove`

说明：

- `speedStdLike` 不要求严格统计学标准差，实现上可用轻量近似指标
- `hasEffectiveMove` 仅表示点击前窗口里是否存在“超过最小位移阈值”的有效移动

### 4.5 有效移动阈值

为避免鼠标轻微抖动污染数据，需要定义最小有效位移。

推荐首版：

- 相邻采样点位移 `< 3px` 视为抖动，不算有效移动

## 五、后端判定方案

### 5.1 新增信号类型

轨迹风控不直接替代现有点击限流，而是新增一个独立低权重信号。

建议新增风险事件：

- `mouse_path_suspicious`

建议分值：

- `+1`

理由：

- 首版信息量有限
- 误伤代价高
- 需要配合现有 `click_rate_limit_hit` 等事件一起看

### 5.2 首版只判“有轨迹但过于程序化”

首版判定必须满足“先有轨迹”：

- `sampleCount` 达到最小值
- `hasEffectiveMove = true`

否则直接视为“信息不足”，不记分。

### 5.2.1 贝塞尔曲线算不算

算，但不是“看见平滑轨迹就算脚本”。

这里的口径必须更严格：

- 真人移动也可能出现平滑轨迹
- 浏览器事件本身就会把真实移动采样成较平滑的折线
- 低采样率下，很多真实轨迹都会显得比原始手势更顺

因此，首版后端不是去识别“它是不是贝塞尔曲线本身”，而是识别：

- 长距离移动过于接近理想直线
- 速度变化过于均匀
- 转向次数过少
- 多次点击前摘要高度重复

也就是说：

- “像贝塞尔”只能算候选异常
- 必须和其他程序化特征叠加，才进入记分候选
- 单独一个“轨迹很顺”不能记分

### 5.3 程序化轨迹候选特征

首版不追求复杂，只做少量阈值组合：

1. 曲折度过低
   - `detourRatio` 过于接近 `1`
   - 说明路径接近完美直线

2. 方向变化过少
   - 长距离移动但 `directionChanges` 很低
   - 更像程序生成直线插值

3. 速度波动过小
   - `speedStdLike` 长期极低
   - 更像匀速数学路径

4. 短时间摘要高度重复
   - 同账号在短窗口内多次点击前摘要相近
   - 更像模板化回放

注意：

- 以上任一条都不单独直接定为脚本
- 首版建议至少命中其中两条，再进入“可记分候选”

### 5.3.1 摘要字段计算口径

为避免前后端理解不一致，首版把摘要字段的含义固定如下。

假设点击前窗口内采样点为：

- `p0, p1, ..., pn-1`
- 每个点为 `(x, y, t)`

#### `sampleCount`

- 直接等于窗口内采样点数量 `n`

#### `windowMs`

- `tn-1 - t0`

若 `n < 2`，则视为信息不足。

#### `distanceTotal`

- 所有相邻点距离之和
- 公式：
  - `sum(sqrt((xi - x(i-1))^2 + (yi - y(i-1))^2))`

#### `distanceStraight`

- 起点到终点的直线距离
- 公式：
  - `sqrt((xn-1 - x0)^2 + (yn-1 - y0)^2)`

#### `detourRatio`

- `distanceTotal / max(distanceStraight, 1)`

解释：

- 越接近 `1`，越像理想直线路径
- 越高，说明绕路越明显

#### `speedAvg`

- `distanceTotal / max(windowMs, 1)`

单位保持前后端一致即可，首版可按 `px/ms`。

#### `speedStdLike`

首版不要求真实标准差，只要求一个轻量“速度波动指标”。

建议算法：

- 先算每段速度 `vi = segmentDistance / segmentTime`
- 再算 `max(v) - min(v)`
- 或者算 `mean(abs(vi - avgV))`

只要前后端约定一致即可。

首版建议采用：

- `mean(abs(vi - avgV))`

理由：

- 比真正标准差更容易实现
- 已足够区分“几乎匀速插值”和“真人有顿挫的移动”

#### `directionChanges`

首版用折线夹角近似，不做复杂曲率。

建议算法：

- 对每两个连续线段求方向向量
- 计算相邻两段夹角是否超过阈值
- 超过阈值则记一次方向变化

推荐阈值：

- `15°`

也就是：

- 微小抖动不算转向
- 明显折返、修正、微调整算一次转向

#### `pauseCount`

建议算法：

- 连续相邻采样点位移小于最小有效位移阈值
- 且连续持续时间超过 `100ms`
- 记为一次停顿

注意：

- 停顿本身不是负信号
- 真人轨迹通常会带停顿或修正
- 反而“完全不停、全程均速”更可疑

#### `hasEffectiveMove`

窗口内只要存在任一段：

- `segmentDistance >= minEffectiveMovePx`

就记为 `true`。

推荐首版：

- `minEffectiveMovePx = 3`

### 5.3.2 首版保守阈值建议

以下不是最终硬编码，只是文档级首版建议值，供实现和观察期使用：

- `minSamples >= 8`
- `distanceStraight >= 24px` 才允许进入“程序化轨迹判断”
- `detourRatio <= 1.08` 视为“过直”
- `directionChanges <= 1` 视为“转向过少”
- `speedStdLike <= 0.12 * speedAvg` 视为“速度波动过小”

这里特意加入 `distanceStraight >= 24px` 的门槛，原因是：

- 很短的小移动本来就容易看起来像直线
- 如果位移不够长，去判断“像不像贝塞尔/直线插值”意义不大

### 5.3.3 首版候选判定逻辑

首版后端建议按下面顺序判断：

1. `sampleCount < minSamples`
   - 直接返回“信息不足”
2. `hasEffectiveMove = false`
   - 直接返回“无有效轨迹，不记分”
3. `distanceStraight < 24px`
   - 直接返回“轨迹过短，不做程序化判断”
4. 计算三类程序化子信号：
   - `tooStraight = detourRatio <= 1.08`
   - `tooFewTurns = directionChanges <= 1`
   - `tooSmooth = speedStdLike <= 0.12 * speedAvg`
5. 若三项中命中至少两项
   - 标记为 `mousePathCandidate = true`
6. 再进入组合风控层，决定是否真正记分

### 5.4 与现有风控组合

推荐首版真正记分的触发条件：

- 当前点击轨迹摘要命中“程序化轨迹候选”
- 且同时满足以下至少一项：
  - 同时命中现有点击异常探测
  - 最近短窗口内重复命中程序化轨迹
  - 当前账号已有未过期风险分

这样做的目的：

- 不让轨迹成为单点误伤来源
- 只在“本来就有别的异常迹象”时提高分数

### 5.4.1 后端处理流程

建议后端把轨迹风控放在实时点击主链路里，但严格限制为轻量步骤。

推荐处理顺序：

1. 正常执行现有点击请求解析
2. 先跑现有账号封禁检查
3. 先跑现有点击异常探测
4. 再读取本次点击附带的轨迹摘要
5. 计算 `mousePathCandidate`
6. 结合本次点击异常探测结果、账号现有风险状态、近期重复命中状态
7. 决定是否记录 `mouse_path_suspicious`
8. 点击主流程继续成功返回，不因这条轨迹规则直接拦截

关键点：

- 轨迹风控只负责“记分”
- 不负责“拒绝本次点击”
- 即使命中，也不返回新的同步错误码

### 5.4.2 后端重复摘要判断

文档里“重复摘要”必须明确成可实现口径。

首版不保存完整轨迹，只保存一个轻量摘要签名。

建议做法：

- 从以下字段生成一个粗粒度签名：
  - `sampleCount` 分桶
  - `distanceStraight` 分桶
  - `detourRatio` 分桶
  - `directionChanges` 分桶
  - `speedStdLike/speedAvg` 分桶

例如：

- `samples:8-12`
- `straight:24-40`
- `detour:1.00-1.08`
- `turns:0-1`
- `smooth:0.00-0.12`

把这些桶值拼成一个短字符串，作为最近一次轨迹摘要签名。

后端只需要：

- 记录“该账号最近一次程序化候选签名”
- 记录对应过期时间

若在短窗口内再次出现相同或高度相邻的签名，可视为“重复摘要命中”。

### 5.4.3 建议 Redis 状态

如果进入实现，后端建议新增两类轻量 Redis 状态：

1. 最近一次轨迹候选摘要签名
   - 用于判断“短时间重复模板”

2. 轨迹记分抑制标记
   - 用于防止同一段异常状态连续刷分

推荐 TTL：

- 摘要签名状态：`60s`
- 记分抑制标记：`60s`

这样可以保证：

- 不需要长历史
- 不增加重型存储
- 与现有 `anti_script.score_window` 解耦

### 5.5 重复记分抑制

轨迹风控必须复用现有“窗口内去重/边沿触发”思想。

推荐规则：

- 同一账号在短窗口内连续命中相似轨迹，不每次都加分
- 只在从“未命中”切换到“命中状态”时记录一次

首版建议：

- 轨迹命中抑制窗口 `60s`

这和总积分过期窗口不是一回事：

- `60s` 只负责避免短时间连续刷分
- `anti_script.score_window` 负责总分何时自然过期

### 5.5.1 一次完整记分条件

为了把行为说死，首版“真正写入 `mouse_path_suspicious`”建议同时满足：

1. 本次摘要先满足 `mousePathCandidate = true`
2. 以下三项至少满足一项：
   - 本次同时命中现有 `click_rate_limit_hit` 候选
   - 近 `60s` 内命中过相同轨迹摘要签名
   - 当前账号已存在未过期风险分
3. 当前账号不存在轨迹记分抑制标记

满足后：

- 写入一次 `mouse_path_suspicious`
- 同时写入 `60s` 抑制标记

不满足则：

- 只保留候选态，不记分

## 六、性能约束

### 6.1 前端性能约束

前端必须满足：

- 常驻采样为 O(1) 写入环形缓冲区
- 点击时仅计算一次小摘要
- 不做点击后全量轨迹序列化
- 不做长历史比对

按 `20Hz * 1500ms` 估算：

- 点击前窗口大约只有 `30` 个采样点

这个量级足够轻。

### 6.2 网络开销约束

每次点击额外上传的应是小 JSON 摘要，而不是数组。

首版目标：

- 单次额外负载控制在数十到数百字节量级

这样不会明显推高 WebSocket 点击载荷。

### 6.3 后端性能约束

后端热路径只允许：

- 读取本次摘要
- 做几个阈值判断
- 最多写入少量 Redis 状态

明确不允许：

- 点击时把完整轨迹落库
- 点击时跑复杂统计建模
- 点击时扫描长历史事件

## 七、接入边界

### 7.1 建议接入点

接入实时点击主链路即可：

- 前端：`frontend/src/utils/realtimeTransport.js`
- 后端：`backend/internal/httpapi/realtime_socket.go`
- 点击执行入口：`backend/internal/httpapi/realtime_helpers.go`

### 7.2 不影响的链路

首版不接入：

- 离页挂机
- 非实时点击接口
- 登录接口
- 购买体力接口

原因很简单：

- 轨迹只对“浏览器实时手点”有意义

## 八、配置建议

建议把轨迹风控作为 `anti_script` 下的新分组配置：

- `mouse_path.enabled`
- `mouse_path.sample_hz`
- `mouse_path.window_ms`
- `mouse_path.min_samples`
- `mouse_path.min_effective_move_px`
- `mouse_path.repeat_suppress_seconds`
- `mouse_path.points.suspicious`

推荐首版默认值：

- `enabled: false`
- `sample_hz: 20`
- `window_ms: 1500`
- `min_samples: 8`
- `min_effective_move_px: 3`
- `repeat_suppress_seconds: 60`
- `points.suspicious: 1`

默认关闭的原因：

- 先完成实现和验收
- 通过小范围观察后再开

## 九、上线策略

建议分三阶段：

### 9.1 第一阶段：只采样不上分

- 前端上传摘要
- 后端只打日志或写调试统计
- 不写 `anti_script` 分数

目的：

- 验证真实玩家摘要分布
- 看触摸板、定点狂点是否大量出现“假阳性”

### 9.2 第二阶段：低权重记分

- 开启 `mouse_path_suspicious = 1`
- 只在组合条件满足时记分

目的：

- 验证它是否能给现有风控提供增量价值

### 9.3 第三阶段：按效果再调阈值

- 看后台风险事件历史
- 再决定是否要加强“重复摘要”规则

首版不建议直接提高到高权重。

## 十、已知局限

本方案明确拦不住以下情况：

- 录制真人轨迹后回放的高级脚本
- 专门模拟抖动、停顿和变速的自动化
- 多浏览器隔离环境
- 无需鼠标移动、只靠键盘或辅助输入完成的少数场景

因此，这套方案的正确定位是：

> 给现有 `anti_script` 增加一个低成本、低误伤、能抓低级自动化的辅助信号，而不是提供“同机识别”或“绝对脚本判定”。

## 十一、评审重点

本方案在进入实施前，建议重点 review 以下问题：

1. `20Hz / 1500ms / +1 分` 这个保守参数是否接受
2. “无轨迹不记分”这一误伤控制口径是否维持不变
3. 首版是否先走“只采样不上分”
4. 摘要字段是否还需要进一步收缩
5. 是否要把“重复摘要”判断延后到第二阶段再做

## 十二、当前建议

如果按最小风险落地，建议顺序是：

1. 先做 `20Hz` 采样和摘要上传
2. 先不上分，只观测日志
3. 观察一轮真实数据后，再决定是否启用 `mouse_path_suspicious`

这比一开始就把轨迹直接接进处罚逻辑更稳。
