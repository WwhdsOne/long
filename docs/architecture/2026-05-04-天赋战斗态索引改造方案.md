# 天赋战斗态索引改造方案

> 状态：已审核，待实现
> 日期：2026-05-04
> 结论：当前天赋战斗态查询的热点问题来自 `SCAN prefix*suffix` 全键空间遍历；首版应保留现有主键结构，新增 `bossID -> nickname Set` 索引，读路径改为 `SMEMBERS`，并在兼容期保留一次回退补建能力。

## 一、问题确认

当前慢点位已经明确：

- Redis 查询使用 `SCAN`
- 模式为 `player:talent_state:*:{bossID}`
- 查询目标是“找出某个 Boss 下所有存在战斗态的玩家昵称”

仓库中对应实现位于 `backend/internal/core/store.go` 的 `listTalentCombatStateNicknames`：

- 前缀：`{namespace}player:talent_state:`
- 后缀：`:{bossID}`
- 匹配模式：`prefix + \"*\" + suffix`

这条路径的结构性问题不是 `COUNT` 太小，而是：

- 它依赖 `SCAN + MATCH` 扫描全库
- 目标维度是按 `bossID` 查，但现有 key 排布是按 `nickname -> bossID`
- 只要键规模继续增长，这个热点就会持续恶化

因此，调 `COUNT` 只能缓解，不解决复杂度问题。

## 二、改造目标

首版目标只做一件事：

- 把“按 `bossID` 找昵称列表”的复杂度，从“扫全库”改成“直接读索引”

同时保持这些约束：

- 不重排现有 `player:talent_state:{nickname}:{bossID}` 主键格式
- 不做历史全量迁移脚本前置依赖
- 不引入 RediSearch、额外存储模块或复杂二级索引系统
- 不为兼容旧逻辑长期保留双读分支

## 三、索引设计

### 3.1 新增 Redis 索引键

新增键：

```text
{namespace}talent_state:boss_idx:{bossID} -> Set<nickname>
```

含义：

- 这个 Set 只表达“哪些昵称当前在这个 Boss 下存在天赋战斗态”
- 不承载数值真值
- 真值仍保留在原有 `player:talent_state:{nickname}:{bossID}` 键

这样拆分后：

- 主状态读写语义不变
- 反向查询从 `bossID` 命中昵称列表变成 O(1) 索引读

### 3.2 为什么用 Set，不用 Sorted Set / Hash / 重排 key

Set 足够，因为当前查询只需要“存在集合”，不需要：

- 排序
- score 范围过滤
- 多字段检索

不选其他方案的原因：

- `Sorted Set` 没有新增价值，只增加写复杂度
- `Hash` 不适合表达纯成员集合
- 重排主键虽然也能改善匹配，但需要迁移所有读写路径，风险更高

因此首版选 Set 是最小改动解法。

## 四、读写路径调整

### 4.1 读路径

`listTalentCombatStateNicknames` 改为：

1. 先读 `talent_state:boss_idx:{bossID}`
2. 使用 `SMEMBERS` 直接拿到昵称列表
3. 对昵称做去空值与必要去重
4. 返回给后续天赋 tick / 结算流程

兼容期兜底：

- 如果索引键不存在或返回空，而当前流程判断理论上不应为空，则允许回退一次旧 `SCAN`
- 回退仅作为过渡期保护，不作为长期常态路径

### 4.2 写路径

凡是会创建、保留、删除玩家天赋战斗态的路径，都必须同步维护索引。

至少包括：

- 首次生成 `player:talent_state:{nickname}:{bossID}` 时执行 `SADD`
- 删除该战斗态时执行 `SREM`
- Boss 结束统一清理战斗态时同步清理对应索引成员
- 重置、逃逸、异常清理等所有“状态消失”路径都要落 `SREM`

原则：

- 只要主状态存在，索引成员就必须存在
- 只要主状态被删，索引成员就必须被删

### 4.3 自愈补建

兼容期允许保留一个轻量自愈机制：

- 读索引未命中时，回退旧 `SCAN`
- 如果 `SCAN` 扫到有效昵称，则立刻补建 `boss_idx`

这个机制的目的不是长期容错，而是：

- 避免上线瞬间要求先跑全量迁移
- 让旧数据在真实访问中按需回填

当线上确认索引稳定后，应删掉回退分支，避免技术债长期保留。

## 五、一致性与清理要求

### 5.1 脏索引容忍

即使索引成员残留，读流程也不能直接信任它一定存在主状态。

因此后续遍历昵称读取具体战斗态时：

- 如果主状态不存在，直接跳过
- 并应顺手把该昵称从索引中移除，做惰性清理

### 5.2 Boss 生命周期清理

Boss 结束后，不能只删玩家战斗态，不删索引。

必须保证：

- 玩家战斗态删除完成
- `talent_state:boss_idx:{bossID}` 最终被清空或整体删除

否则历史 Boss 会留下无意义索引集合。

### 5.3 不做长期双模型

这个方案是“兼容上线”，不是“双模型常驻”。

阶段边界应明确：

- 第一阶段：双写 + 回退补建
- 第二阶段：确认线上索引完整
- 第三阶段：删除回退 `SCAN` 主路径

如果长期保留旧 `SCAN`，问题只是被隐藏，不是被解决。

## 六、实施顺序

建议顺序：

1. 先加索引 key 生成函数与索引维护辅助函数
2. 把所有战斗态写入 / 删除路径补齐 `SADD/SREM`
3. 把 `listTalentCombatStateNicknames` 改为优先读索引
4. 保留过渡期 `SCAN` 回退与按需补建
5. 增加测试覆盖
6. 观察线上一段时间后移除回退逻辑

## 七、验收标准

功能验收：

- 同一 Boss 下存在战斗态的玩家，都能被索引读出
- 清理战斗态后，索引不残留有效成员
- Boss 结束后，相关索引被清空或删除

性能验收：

- 该热点查询不再出现 `SCAN prefix*suffix`
- Redis 慢查询中该路径显著下降
- 同数据规模下，查询耗时从“随键规模增长”变为“与该 Boss 参与人数近似相关”

## 八、不选方案说明

当前不选这些方案：

- 只调 `SCAN COUNT`
  - 只能调节单次返回批量，不能改变全库扫描本质
- 重排主键顺序
  - 改动范围更大，首版没有必要
- Lua 包裹多次 `SCAN`
  - 只能减少网络往返，不能解决 Redis 端扫描成本
- RediSearch
  - 对当前问题过重，维护成本过高

因此，Set 反向索引是这次改造的首选路线。
