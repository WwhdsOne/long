# gRPC + Envoy 点击链路评估方案

> 状态：待评估，未实施
> 日期：2026-05-04
> 结论：当前不应直接上 `gRPC + Envoy` 替换现有点击链路；应先在现有 WebSocket 链路内完成批量化与观测补齐，再用压测数据决定是否进入 Envoy 或 gRPC 阶段。

## 一、当前实现事实

基于当前仓库代码，点击链路已经不是“前端每次点一下都发一个独立 HTTP 请求”。

- 前端实时点击入口走 WebSocket，服务端处理在 `backend/internal/httpapi/realtime_socket.go`
- 客户端消息类型包含 `click`
- 点击消息最终进入 `executeButtonClick`
- `executeButtonClick` 会先做限流，再调用 `Store.ClickButton`
- `Store.ClickButton` 最终通过 Redis Lua 脚本合并执行 `HINCRBY`、`ZINCRBY`、`INCRBY` 等写操作
- 当前限流在应用内完成，按 `ip:<clientID>` 和 `nickname:<nickname>` 双键检查

这意味着：

- Redis 往返已经做过一次合并，瓶颈不在“单次点击要打很多条 Redis 命令”
- 当前链路的主要问题更可能是“消息粒度太细”“每条点击都要走一遍会话层与业务层”“缺少足够的链路指标”
- 如果没有新的压测数据支撑，直接引入 `gRPC + Envoy` 风险高于收益

## 二、为什么现在不建议直接切 gRPC + Envoy

### 2.1 gRPC 不能替代现有浏览器接入现实

浏览器端现在已经稳定运行在 WebSocket / HTTP 语义上。

- 如果做端到端 gRPC，浏览器侧实际要走 `gRPC-Web + Envoy`
- 这不是“把 JSON 改成 protobuf”这么简单，而是要同时引入：
  - 新协议定义
  - 新网关
  - 前端接入层改造
  - 新的调试与排障路径

首版收益未必覆盖迁移成本。

### 2.2 现有痛点更像“批量问题”，不是“协议问题”

当前代码已经支持 `comboCount` 传入 `ClickButton`。

这说明服务端存储层天然支持“单条消息代表多次点击”的聚合语义，只是现有实时协议还没有把它正式收敛成可观测、可控的批量点击模型。

如果先在当前 WebSocket 协议上增加批量消息：

- 可以直接复用现有鉴权、限流、变更发布、实时 ack 链路
- 可以先验证“批量化本身能带来多大收益”
- 可以避免在没有收益基线时过早引入新基础设施

### 2.3 Envoy 的主要价值在治理，不在首刀优化

Envoy 的真实价值主要是：

- 连接治理
- 限流与熔断
- 观测指标
- 上游负载均衡
- 后续多服务协议桥接

如果当前只有单个 Hertz 服务承接点击主路径，那么 Envoy 首先解决的是“治理能力不足”，不是“点击一定慢”。

因此 Envoy 可以做，但应该建立在：

- 已经补齐现有链路指标
- 已确认单机应用层限流与 WebSocket 批量仍不能满足目标
- 已经存在多实例、多服务或跨协议拆分需求

## 三、建议的分阶段路线

### 3.1 第一阶段：只改现有 WebSocket 点击协议

目标：最低风险拿收益。

建议新增消息能力：

- 新增 `batch_click` 消息类型
- 单条消息允许携带：
  - `slug`
  - `delta` 或等价的批量点击次数
  - `requestId`
- 服务端收到后只做一次限流检查、一次 `Store.ClickButton`
- 返回一次聚合后的 `click_ack`

同时保留现有 `click`：

- 旧客户端继续可用
- 新客户端逐步切到 `batch_click`
- 服务端内部可以统一收敛到同一个执行函数

阶段目标：

- 降低每单位点击的会话层消息数
- 降低 JSON 编解码次数
- 降低应用层限流与业务判定次数
- 为后续 gRPC 评估建立对照基线

### 3.2 第二阶段：补齐点击链路观测

在决定是否进入 Envoy / gRPC 之前，先补齐这些指标：

- WebSocket 入站点击消息 QPS
- 平均批量大小
- `Store.ClickButton` 延迟分布
- Lua 执行耗时分布
- 点击成功率 / 拒绝率 / 限流率
- 变更发布耗时
- `click_ack` 返回延迟

验收口径至少包含：

- p50 / p95 / p99
- 服务端 CPU
- Redis ops/sec
- Redis command latency
- 出入站带宽

没有这组数据，不应该讨论 gRPC 是否“值得上”。

### 3.3 第三阶段：只在需要时引入 Envoy

进入条件：

- 已完成第一阶段批量化
- 已完成第二阶段链路观测
- 数据证明当前瓶颈仍在接入层或治理层

此时 Envoy 的定位应明确为“网关与治理层”，而不是“性能银弹”。

首版只建议承担这些职责：

- WebSocket / HTTP 接入统一入口
- 连接与超时治理
- 限流补强
- 指标采集
- 多实例上游负载均衡

不建议首版同时做：

- 端到端协议整体迁移
- 复杂流量镜像
- 多版本协议并行很久

### 3.4 第四阶段：内部 gRPC 只在服务拆分时进入

如果未来点击链路拆成独立服务，再评估：

- Envoy 入口
- Hertz 网关层
- 内部 gRPC 点击服务

此时 gRPC 才更合理，因为：

- 内部服务间调用更适合强 schema
- protobuf 对服务间高频调用更有意义
- Envoy 的上游治理价值开始兑现

在当前单体服务形态下，内部 gRPC 收益有限。

## 四、推荐结论

推荐路径按顺序执行：

1. 保持现有 Hertz + WebSocket + Redis Lua 主链路不变
2. 增加 `batch_click` 协议并复用现有 `comboCount` 语义
3. 补齐点击链路指标与压测基线
4. 如果数据仍显示接入层瓶颈明显，再加 Envoy
5. 只有在服务拆分或跨语言协作需求明确后，再引入内部 gRPC

不推荐的路径：

- 现在就把点击主链路直接切到 `gRPC-Web + Envoy`
- 现在就把单体内部改成 gRPC，只为“理论上更快”

## 五、评估验收标准

只有满足以下任一条件，才继续进入 Envoy / gRPC 设计：

- 第一阶段批量化后，点击链路 p95 仍明显超标
- 第一阶段批量化后，服务端 CPU 仍被点击流量主导
- 多实例部署下，当前应用内限流与连接治理开始明显失真
- 业务决定把点击链路拆成独立服务

如果第一阶段已经把指标打到目标区间，则本议题到“WebSocket 批量化 + 指标补齐”就应收口，不继续扩复杂度。
