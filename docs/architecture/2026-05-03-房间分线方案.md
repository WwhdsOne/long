# 房间分线方案

> 状态：待审核
> 日期：2026-05-03

## 一、背景与问题

当前全服共享单一 Boss，所有玩家在同一个"世界"战斗。随着游戏运营，玩家练度差距逐渐拉大：

- 高练度玩家：秒伤 6 位数（10万+），Boss 很快被击杀
- 低练度玩家：全程 5 位数都难打出，体验极差，奖励分配也处于劣势

这导致低练度玩家缺乏参与感，可能造成流失。

## 二、需求

| 需求 | 说明 |
|------|------|
| 固定频道 | 预设 N 个频道，玩家可自由选择加入 |
| Boss 完全独立 | 每个频道有独立的 Boss 实例、血量、击杀奖励 |
| 切换冷却 | 切换频道后需等待一定时间才能再次切换 |
| 可配置 | 频道数量和冷却时间通过 Consul 配置动态调整 |

## 三、当前架构

### 3.1 数据结构

```
Boss 状态：    {ns}boss:current              → 全局唯一 Boss（Hash）
伤害排行榜：  {ns}boss:{bossID}:damage      → 按 Boss ID 隔离（Sorted Set）
玩家数据：    {ns}user:{nickname}           → 按昵称索引（Hash）
实时推送：    {ns}events                    → 单一 Pub/Sub channel
```

### 3.2 推送机制

```
业务变更 → StateChange → Redis Pub/Sub → Dispatcher → Hub → SSE/WebSocket → 浏览器
```

- Hub 是全局单例，`clients map[*subscriber]struct{}` 无分区
- `BroadcastPublic` 向所有 clients 广播同一个 Snapshot
- `BroadcastUser` 按 nickname 精确匹配，无房间概念

### 3.3 关键事实

- 只有一个 `boss:current` key，所有玩家共享
- Lua 脚本（`bossClickScript`）操作全局 key
- 玩家数据按 nickname 索引，无房间维度
- Boss 由管理后台手动激活，系统启动时不主动创建

## 四、方案设计

### 4.1 数据隔离

```
旧版本：
  {ns}boss:current                    → 全局唯一 Boss

新版本：
  {ns}boss:current:{roomID}           → 每个频道独立 Boss
  {ns}player:room:{nickname}          → 玩家当前所在频道
  {ns}player:room:cd:{nickname}       → 频道切换冷却到期时间戳
```

Boss 模板池（`boss:pool:*`）保持全局共享，激活时按房间创建独立实例。

### 4.2 Hub 推送隔离

```
旧版本：
  Subscribe(nickname)                 → 全局订阅
  BroadcastPublic(snapshot)           → 广播给所有人

新版本：
  Subscribe(nickname, roomID)         → 按房间订阅
  BroadcastRoom(roomID, snapshot)     → 只广播给该房间的连接
  BroadcastPublic(snapshot)           → 保留为全局广播（公告等）
```

subscriber 增加 `roomID` 字段，`broadcastOnlineCountLocked` 按房间统计。

### 4.3 Redis Pub/Sub

保持单一 channel（`{ns}events`），`StateChange` 增加 `RoomID` 字段。进程内根据 `RoomID` 过滤，只推送给对应房间的连接。

### 4.4 Lua 脚本改造

Boss 点击脚本的 `KEYS` 参数从全局 key 改为房间 key：

```
旧：KEYS[1] = {ns}boss:current
新：KEYS[1] = {ns}boss:current:{roomID}
```

### 4.5 Consul 配置扩展

```yaml
room:
  enabled: true
  ids: ["1", "2", "3"]
  default_room: "1"
  switch_cooldown_seconds: 300
```

### 4.6 API 设计

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/rooms` | GET | 获取频道列表、在线人数、当前所在频道 |
| `/api/rooms/join` | POST | 切换频道（带冷却检查） |
| `/api/events?room=2` | SSE | 带房间参数的 SSE 连接 |
| `/api/ws` | WS | `hello` 消息增加 `roomID` 字段 |

### 4.7 前端改造

- `realtimeTransport.js`：连接支持 roomID 参数
- 新增 `RoomSelector.vue`：频道选择组件（显示频道列表、在线人数、冷却倒计时）
- `publicPageState.js`：新增 `currentRoom` 状态管理
- `BattlePage.vue`：顶部显示当前频道，集成 RoomSelector

## 五、迁移方案

### 5.1 迁移目标

- **零停机**：上线过程不中断服务
- **数据不丢失**：现有玩家数据完整保留
- **向后兼容**：旧客户端可正常访问（默认房间）
- **可回滚**：出问题可快速回退

### 5.2 迁移策略：首次启动兼容层

不写独立迁移脚本，在新版本代码中加入兼容层，首次启动时自动迁移。

#### Boss 数据迁移

```go
// Store 初始化时检查
func (s *Store) migrateBossToRoom(ctx context.Context) error {
    // 1. 检查是否存在旧的 boss:current
    oldBoss, err := s.currentBoss(ctx, "")
    if err != nil || oldBoss == nil {
        return nil  // 没有旧数据，无需迁移
    }

    // 2. 检查新版本的 boss:current:1 是否已存在
    newBoss, _ := s.currentBoss(ctx, "1")
    if newBoss != nil {
        return nil  // 已迁移过
    }

    // 3. 复制到默认房间
    return s.setCurrentBoss(ctx, oldBoss, "1")
}
```

#### 玩家房间惰性初始化

```go
// 玩家首次连接时，如果没有房间记录，默认分配到房间 "1"
func (s *Store) ensurePlayerRoom(ctx context.Context, nickname string) (string, error) {
    roomID, err := s.getPlayerRoom(ctx, nickname)
    if err != nil || roomID == "" {
        roomID = "1"
        s.setPlayerRoom(ctx, nickname, roomID)
    }
    return roomID, nil
}
```

#### 旧客户端兼容

```
旧客户端连接 → 不带 roomID 参数 → 服务端默认分配到房间 "1"
```

- SSE: `/api/events` → 默认房间 "1"
- WS: `hello` 消息无 `roomID` → 默认房间 "1"
- 点击请求：无 `roomID` → 使用当前房间 "1"

### 5.3 上线步骤

#### 阶段 1：准备

1. 更新 Consul 配置，新增 `room` 配置项
2. 部署新版本代码（包含兼容层）

#### 阶段 2：自动迁移

1. 服务首次启动，检测到旧 `boss:current` → 复制到 `boss:current:1`
2. 玩家首次连接时自动分配到房间 "1"

#### 阶段 3：验证

1. 检查 `boss:current:1` 存在且数据正确
2. 检查所有在线玩家都在房间 "1"
3. 测试房间切换 API
4. 测试多房间 Boss 独立性

#### 阶段 4：清理（可选，延迟执行）

1. 删除旧 key `boss:current`（已迁移到 `boss:current:1`）
2. 调整监控面板支持房间维度

### 5.4 回滚方案

#### 快速回滚

1. 回滚代码：部署旧版本
2. 回滚配置：删除 Consul 中的 `room` 配置
3. 数据保留：新产生的 `boss:current:1` 数据保留，不影响旧版本

#### 数据兼容

旧版本代码会忽略 `boss:current:1`，只读取 `boss:current`：
- 旧版本继续使用 `boss:current`（未被修改）
- 新版本使用 `boss:current:1`（新创建）

## 七、风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 迁移脚本失败 | Boss 数据丢失 | 迁移前备份 Redis |
| 旧客户端不兼容 | 无法攻击 Boss | 默认房间兜底 |
| 配置错误 | 频道数量异常 | 配置校验 |
| 冷却时间过短 | 频繁切换 | 可配置调整 |
| Lua 脚本改动 | 点击异常 | 充分测试 |

## 八、实施顺序

1. 后端核心：StateChange + Hub + Dispatcher
2. 后端数据：Boss 隔离 + Lua 脚本 + 房间管理
3. 后端 API：房间列表 + 切换接口
4. 后端配置：Consul 配置扩展 + 兼容层
5. 前端传输：realtimeTransport 支持 roomID
6. 前端 UI：RoomSelector + BattlePage 集成
7. 测试验证
