# 房间分线方案

> 状态：已审核，待实现
> 日期：2026-05-03
> 结论：可做，但首版应收敛为“Boss 战斗与循环控制按房间隔离”；允许停机维护；前端永远只有最新版本；不承诺无损回滚。

## 一、审核结论

这份方案现在可以收敛为一套更明确的模型：

- 房间维度不能只加在 `Hub / PubSub` 层。当前首屏状态、Boss 资源、SSE 初始快照、WebSocket `snapshot`、`/api/battle/state` 都直接读取全局状态；如果只改广播层，会出现“首屏看的是全局 Boss，增量推送看的是房间 Boss”的错位。
- 既然已经定义“Boss 从属于循环队列，循环队列再挂到房间”，那 `boss:current` 和 `boss:cycle` 都必须一起房间化，不能只拆当前 Boss，不拆循环状态。
- 当前项目允许停机维护，且前端永远只有最新版本，因此不需要保留“旧前端兼容”“运行时双读双写”“旧协议继续可用”这类复杂分支。
- Mongo 冷数据不能只补 `room_id`，还应同时补 `queue_id`，并把索引一起补齐；历史文档可以统一回填为默认房间 `1`、默认队列 `1`。

## 二、目标与范围

### 2.1 首版目标

- 预置固定房间 ID，玩家可切换当前房间
- 每个房间拥有独立的当前 Boss、血量、伤害榜和击杀奖励
- 每个房间拥有独立的循环控制状态；Boss 从属于该房间当前绑定的循环队列
- 某个房间开启循环后，如果该房间当前没有活动 Boss，则立即从该房间的循环队列拉起一只 Boss
- 每个房间保留原有后台能力：跳过当前 Boss、开启循环、关闭循环、停用当前 Boss、手动激活下一只 Boss
- 切房受冷却限制
- 未激活且未开启循环的房间不对玩家开放，前端房间按钮直接禁用
- 离页挂机按“挂机开始时玩家当前房间”结算，不跨房间漂移

### 2.2 首版不做

- 不把 `totalVotes` 和点击总榜拆成按房间统计；它们继续全服共享
- 不做配置热生效；Consul 发布后按现有机制重启服务生效
- 不做分线期间 Boss 进度的无损回滚
- 不做自动删房迁移；房间缩容只做配置回退与人工清理
- 不保留多版本前端或旧实时协议兼容逻辑

## 三、当前架构约束

结合当前仓库代码，分线不是单点改动，至少有这些现实约束：

- `backend/internal/core/store.go` 中 `GetSnapshot`、`GetUserState`、`GetState`、`GetBossResources`、`ClickBossPart` 都围绕单一 `boss:current` 运行。
- `backend/internal/core/boss_pool.go` 中 `boss:cycle`、循环队列、下一只 Boss 激活逻辑也都是全局唯一。
- `backend/internal/events/cache.go` 只有一份公共快照缓存、一份 Boss 资源缓存，用户缓存也只按 `nickname` 建索引。
- `backend/internal/events/hub.go` 的订阅没有房间维度，SSE 初始快照同样直接读取全局状态。
- `backend/internal/httpapi/realtime_socket.go` 的 `snapshot` 和点击后推送都走全局 `StateView`。
- `backend/internal/httpapi/battle_routes.go` 的 `/api/battle/state` 也是全局读取。
- `backend/internal/mongostore/boss_history.go`、`domain_events.go`、`admin_audit.go` 当前都没有 `room_id` / `queue_id` 字段与索引。
- `backend/internal/config/config.go` 的 Consul watcher 在配置变化时会主动退出进程。

因此，本方案必须把房间维度推进到“状态读取入口”和“循环控制入口”，不能只停留在广播过滤。

## 四、修订后的方案

### 4.1 房间真值与循环真值

房间归属以后端状态为准，不以后端被动相信客户端传参为准。

新增或改造后的 Redis 键：

```text
{ns}player:room:{nickname}        -> 玩家当前房间
{ns}player:room:cd:{nickname}     -> 切房冷却到期时间戳
{ns}boss:current:{roomID}         -> 房间当前 Boss
{ns}boss:cycle:{roomID}           -> 房间循环控制状态（queue_id / enabled / queue）
```

约束：

- 玩家有昵称时，服务端先按昵称解析当前房间，再读取 Boss 相关状态
- 没有房间记录时，惰性落到 `default_room`
- 房间配置缩容时，如果玩家当前房间已不在配置中，读取时回落到 `default_room`
- 玩家切房时，服务端校验目标房间是否合法、冷却是否到期、房间是否可加入
- 首版每个房间只挂一份活跃循环控制器；为减少歧义，`boss:cycle:{roomID}` 中应显式保存 `queue_id`，首版默认取与 `roomID` 相同的值

### 4.2 Boss 与循环队列模型

首版采用“每个房间一份循环控制器”的实现模型：

- `boss:current:{roomID}` 表示该房间当前正在攻坚的 Boss
- `boss:cycle:{roomID}` 表示该房间当前绑定的循环队列与开关状态
- 逻辑上 Boss 从属于 `queue_id`，而 `queue_id` 归当前房间控制
- 房间开启循环后，如果该房间当前没有活动 Boss，则立即按该房间队列拉起下一只 Boss
- 房间关闭循环后，不再自动拉起下一只 Boss
- 房间跳过当前 Boss、停用当前 Boss、手动激活下一只 Boss，全部都变成“带 `roomID` 的后台操作”

Boss 隔离规则：

- 当前 Boss：按 `boss:current:{roomID}` 隔离
- Boss 血量、状态、部位、击杀奖励：跟随当前房间 Boss
- Boss 伤害榜：继续沿用 `boss:{bossID}:damage`
- Boss 掉落池：继续沿用 `boss:{bossID}:loot`
- Boss 击杀结算锁：继续沿用 `boss:{bossID}:reward-lock`

这里保留 `boss:{bossID}:damage / loot / reward-lock` 的前提是：

- `bossID` 必须保持全局唯一，不能只做到“单房间内唯一”
- `boss:instance:seq` 必须继续全局共享，不能拆成按房间递增
- 后台手工激活 Boss 时也不能允许不同房间复用相同 `bossID`
- 当前 Mongo `boss_history` 也按 `boss_id` 建唯一索引，因此一旦跨房间复用 `bossID`，会同时打坏 Redis 隔离和历史归档唯一性

全局共享保持不变：

- `totalVotes`
- 点击总榜
- 公告版本
- Boss 模板池 `boss:pool:*`
- Boss 模板索引 `boss:pool:index`
- Boss 实例序列 `boss:instance:seq`

为避免后台、归档和日志失去上下文，建议把 `roomID`、`queueID` 都挂到 `Boss` 结构体上。

### 4.3 Mongo 文档与索引

#### `boss_history`

建议新增字段：

- `room_id`
- `queue_id`

保留现有索引：

- `boss_id` 唯一索引
- `started_at` 倒序索引
- `template_id + started_at` 组合索引

新增索引：

- `room_id + started_at` 倒序索引
- `queue_id + started_at` 倒序索引

#### `domain_events`

建议新增字段：

- `room_id`
- `queue_id`

保留现有索引：

- `created_at` 倒序索引
- `event_type + created_at` 组合索引
- `boss_id + created_at` 组合索引
- `nickname + created_at` 组合索引

新增索引：

- `room_id + created_at` 倒序索引
- `queue_id + created_at` 倒序索引

#### `admin_audit_logs`

建议新增字段：

- `room_id`
- `queue_id`

保留现有索引：

- `created_at` 倒序索引
- `operator + created_at` 组合索引
- `action + created_at` 组合索引

新增索引：

- `room_id + created_at` 倒序索引
- `queue_id + created_at` 倒序索引

实现约束：

- 当前 `boss_history`、`domain_events`、`admin_audit_logs` 都不是把 Go 结构体原样自动写入 Mongo，而是手工做字段映射
- 因此不能只在 `Boss`、`DomainEvent`、`AdminAuditLog` 结构体上新增字段，还必须同步修改 Mongo 文档结构、写入映射、读取映射和索引定义
- 首版不需要单独新建“队列历史集合”；`room_id` 与 `queue_id` 写进现有三张集合就足够支撑检索与审计

### 4.4 读取链路必须前置房间

所有依赖“当前 Boss”的读取，都必须在进入 Store 时就确定房间。

至少包括：

- `/api/battle/state`
- `GET /api/events` 的初始公共态和个人态
- `/api/ws` 首次 `snapshot`
- `GetBossResources`
- `GetUserState` 中的 `MyBossStats`、天赋战斗态、待消费天赋事件

首版建议对玩家链路统一采用：

- 在 Store 内通过 `nickname -> roomID` 解析当前房间

这样做的原因：

- 玩家读写链路更简单
- 不会出现“HTTP 一套房间、SSE 一套房间、WS 一套房间”的分叉
- 前端只需要在切房成功后更新当前房间，不必在每条实时消息里重复携带房间真值

后台接口例外：

- 后台 Boss 与循环控制接口应显式传 `roomID`
- 因为后台操作不绑定普通玩家昵称

挂机链路也必须前置房间：

- 首版将“挂机开始时玩家当前房间”视为挂机地
- 建议在挂机状态里显式记录 `afk_room_id`
- 后续挂机 tick、击杀结算、挂机汇总都按这个房间执行
- 如果该房间 Boss 被击杀且该房间循环仍开启，则挂机继续打该房间下一只 Boss
- 如果该房间循环关闭或后续没有活动 Boss，则挂机继续存在，但后续 tick 收益为 0，直到玩家回前台或该房间重新开启循环并出现活动 Boss

### 4.5 实时链路改造

实时层需要从“全局广播”调整为“按房间广播公共态，按昵称+房间广播个人态”。

建议调整为：

```text
Subscribe(nickname, roomID)
BroadcastRoom(roomID, snapshot)
BroadcastUser(nickname, roomID, userState)
```

补充约束：

- `subscriber` 增加 `roomID`
- `Cache` 的公共快照和 Boss 资源缓存改为按 `roomID` 分桶
- 用户缓存不能再只按 `nickname` 建索引；至少要按 `roomID + nickname` 组合缓存，或在切房时显式失效

`online_count` 不建议在首版做成按房间的实时事件协议变更。更简单的做法是：

- 继续保留现有全局在线人数事件
- 房间在线人数通过 `GET /api/rooms` 返回
- 房间人数口径直接定义为“当前正在攻坚该房间 Boss 的玩家数，包含 AFK 玩家”
- 前端文案直接展示为：`当前 x 人正在攻坚 Boss`

### 4.6 状态变更分发

`StateChange` 可以增加可选 `RoomID`，但只在“确实只影响单个房间”的场景下使用，例如：

- Boss 被点击
- 房间内当前 Boss 被停用
- 房间内跳过当前 Boss
- 房间循环被开启 / 关闭
- 房间内 Boss 被击杀并切到下一只 Boss

对于仍然全局共享、但会出现在房间快照里的内容，例如：

- 公告版本变化
- 点击总榜变化

分发器不应偷懒只发一个全局快照，而应对活跃房间逐个刷新快照并广播。原因很简单：快照里虽然混有全局数据，但也同时带有房间 Boss。

### 4.7 配置模型

当前现行口径里，Consul YAML 不再手写真实房间 ID 列表，而是只写房间数量：

```yaml
room:
  enabled: true
  count: 3
  default_room: "1"
  switch_cooldown_seconds: 300
```

约束说明：

- `enabled=false` 仅作为发布窗口与停机维护时的配置开关；首版不要求在同一版本里长期保留一套旧单房间执行路径
- 服务启动时按 `count` 自动生成真实房间 ID：`1..N`
- `default_room` 必须落在 `1..count` 范围内，否则回退到 `1`
- 房间数量与默认房间属于配置事实
- 每个房间的循环开关、当前队列和当前 Boss 属于 Redis 运行时事实，不放在 Consul YAML 里
- 删除房间不做自动数据迁移，只做“停止分配 + 人工清理”
- 房间显示名不再承载在真实 `roomID` 上，而是外置到 Redis 与后台管理页维护

### 4.8 API 设计

新增：

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/rooms` | GET | 返回配置房间列表、当前房间、剩余冷却、各房间在线人数、`joinable`、房间循环状态、当前 `queueId`、当前 Boss 摘要，以及后端统一回退后的 `displayName` |
| `/api/rooms/join` | POST | 切换当前房间，服务端校验目标房间、冷却，以及目标房间当前是否可加入 |

保持不变但改为房间感知：

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/battle/state` | GET | 按玩家当前房间返回 Boss 相关状态 |
| `/api/events` | SSE | 服务端按玩家当前房间推送对应公共态 |
| `/api/ws` | WS | `hello` 仍可保持现有格式，服务端按玩家当前房间绑定订阅 |

后台 Boss / 循环控制接口要求：

- 保留原有“跳过、关闭循环、开启循环、停用 Boss、手动激活下一只 Boss”的能力
- 这些能力全部改成显式带 `roomID` 的后台操作
- 不再存在“全局唯一一套循环控制按钮”的语义

前端房间选择器交互约束：

- 对 `joinable=false` 的房间按钮直接置灰，不允许点击
- 即使前端已置灰，`POST /api/rooms/join` 仍必须做后端二次校验，防止绕过前端直接请求

## 五、迁移与上线策略

### 5.1 迁移原则

- 允许停机维护
- 使用仓库内一次性迁移命令，不做运行时兼容分支
- 不做双读双写，不做旧前端兼容，不做旧协议兼容
- 迁移时保留旧当前 Boss 的 `bossID` 不变
- Mongo 历史文档允许统一回填为默认房间 `1`、默认队列 `1`
- 不承诺分线期间的无损回滚

### 5.2 迁移范围

Redis 迁移：

- 旧 `boss:current` 迁到 `boss:current:1`
- 旧 `boss:cycle` 迁到 `boss:cycle:1`
- `boss:cycle:1` 中补 `queue_id=1`
- 保留旧当前 Boss 的原始 `bossID`

因此这些按 `bossID` 挂载的附属数据无需搬迁：

- `boss:{bossID}:damage`
- `boss:{bossID}:loot`
- `boss:{bossID}:reward-lock`

保持全局不变：

- `boss:instance:seq`
- `boss:pool:index`
- `boss:pool:*`

玩家房间：

- 现有玩家的 `player:room:{nickname}` 可继续惰性初始化到 `default_room`
- 如果维护窗口内需要一次性批量回填，也统一写成房间 `1`

Mongo 回填：

- `boss_history` 旧文档统一回填 `room_id="1"`、`queue_id="1"`
- `domain_events` 旧文档统一回填 `room_id="1"`、`queue_id="1"`
- `admin_audit_logs` 旧文档统一回填 `room_id="1"`、`queue_id="1"`
- 回填时一并创建本方案新增索引

### 5.3 上线步骤

1. 停服务，阻断点击、挂机、后台 Boss / 循环操作。
2. 备份旧 Redis 关键键和 Mongo 三张集合。
3. 运行一次性迁移命令：
   - `boss:current -> boss:current:1`
   - `boss:cycle -> boss:cycle:1`
   - Mongo 三张集合回填 `room_id=1`、`queue_id=1`
   - 创建新增索引
4. 部署支持房间能力的新版本代码。
5. 启动服务并验证：
   - 房间 `1` 继承旧全局当前 Boss
   - 房间 `1` 继承旧全局循环状态与循环队列
   - 原 `bossID` 不变，旧伤害榜 / 掉落池 / 结算锁仍可继续使用
6. 再验证其他房间的新建循环、切房、房间内后台操作是否独立生效。

### 5.4 回滚约束

当前最小实现只支持“代码和配置回退”，不支持“分线期间 Boss 进度无损映射回旧单房间模型”。

也就是说：

- 可以回退版本
- 可以关闭 `room.enabled`
- 但分线启用后新增的多房间 Boss 进度，旧版代码不会自动理解

如果业务要求回滚到旧单房间版本且仍保留房间 `1` 的进度，需要额外设计反向迁移命令，把：

- `boss:current:1 -> boss:current`
- `boss:cycle:1 -> boss:cycle`

这不纳入本次首版方案。

## 七、主要风险

| 风险 | 说明 | 缓解方式 |
|------|------|----------|
| 只改广播层，没改读取层 | 首屏和增量状态会看到不同 Boss | 明确把房间维度推进到 Store / StateView 入口 |
| 只拆 `boss:current`，没拆 `boss:cycle` | 房间 Boss 与房间循环状态会错位 | 当前 Boss 与循环控制必须一起房间化 |
| 迁移漏掉旧 `boss:cycle` | 房间 `1` 会继承当前 Boss，但不会继承旧循环状态 | 一次性迁移命令同时迁 `boss:current` 和 `boss:cycle` |
| 用户缓存仍只按昵称建索引 | 切房后会拿到旧房间个人态 | 用户缓存改为 `roomID + nickname` 或切房时主动失效 |
| 挂机不锁定房间 | 玩家离页后房间收益归属会漂移 | 挂机开始时记录 `afk_room_id`，后续挂机攻击与结算都按该房间执行 |
| 只给 Go 结构体加字段，没同步 Mongo 映射和索引 | 冷数据归档、事件日志、后台审计仍然丢失房间 / 队列上下文 | 显式修改 `mongostore` 文档结构、索引和读写映射 |
| `bossID` 不是全局唯一 | Redis 伤害榜和 Mongo `boss_history` 唯一索引会互相冲突 | 约束所有房间使用全局唯一 Boss 实例 ID |
| 迁移时服务未停机 | 会出现旧 key / 新 key 同时被写入的窗口期 | 明确停机迁移，不做运行时兼容 |
| 房间缩容 | 现有玩家和房间状态无自动迁移 | 首版只做回落默认房间与人工清理 |

## 八、验收标准

实现时至少要满足这些可验证目标：

1. 同一个玩家在房间 `1` 和房间 `2` 看到的 `boss.id`、`currentHp`、伤害榜彼此独立。
2. 房间 `1` 的 Boss 点击不会改动房间 `2` 的 Boss 血量和房间 `2` 的 Boss 伤害榜。
3. 房间 `1` 继承旧全局当前 Boss，且继承旧全局循环状态与循环队列。
4. 房间 `1` 继承后的当前 Boss 保留原 `bossID`，旧的 `damage / loot / reward-lock` 附属 key 继续有效。
5. 每个房间都保留“跳过当前 Boss、开启循环、关闭循环、停用当前 Boss、手动激活下一只 Boss”的原有后台能力，且只影响本房间。
6. `totalVotes` 和点击总榜仍保持全服共享，不因房间切换而被拆分。
7. `/api/battle/state`、`/api/events` 初始态、`/api/ws` 首次 `snapshot` 对同一玩家返回同一房间视角。
8. 切房接口会校验房间是否合法、冷却是否到期，并在成功后影响后续读取与实时推送。
9. 离页挂机会锁定玩家进入挂机态时的当前房间；同一次挂机期间的攻击、击杀和结算不会漂移到其他房间；若该房间循环关闭或没有活动 Boss，则后续挂机 tick 收益为 0。
10. `boss_history`、`domain_events`、`admin_audit_logs` 三张集合的历史文档完成 `room_id=1`、`queue_id=1` 回填，并创建本方案新增索引。

## 九、实施顺序

1. 先定义房间化的 `boss:current` / `boss:cycle` 模型，并写一次性迁移命令。
2. 再改 `Store / boss_pool / admin`，让 Boss 与循环控制都按房间运行。
3. 再补 `Boss / DomainEvent / AdminAuditLog` 的 `roomID` / `queueID` 字段，以及 Mongo 映射与索引。
4. 再改 `/api/battle/state`、SSE、WebSocket 的首屏读取，让首屏状态先正确。
5. 再改 `Hub / Cache / Dispatcher`，把增量推送切到按房间广播。
6. 再补房间列表、切房接口和后台房间维度控制。
7. 最后接前端房间选择 UI，并补齐迁移与回归测试。
