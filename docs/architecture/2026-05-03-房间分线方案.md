# 房间分线方案

> 状态：已审核，待实现
> 日期：2026-05-03
> 结论：可做，但首版应收敛为“Boss 战斗按房间隔离”，不承诺零停机和无损回滚。

## 一、审核结论

这份方案原始思路是对的，但有四个地方需要先收紧，否则实现时会自相矛盾：

- 房间维度不能只加在 `Hub / PubSub` 层。当前首屏状态、Boss 资源、SSE 初始快照、WebSocket `snapshot`、`/api/battle/state` 都直接读取全局状态；如果只改广播层，会出现“首屏看的是全局 Boss，增量推送看的是房间 Boss”的错位。
- 房间归属必须以后端保存状态为准，不能把 `roomID` 当成客户端可以随意指定的事实来源，否则切房冷却、读写一致性和兼容旧客户端都会变复杂。
- 当前 Consul 配置变更会触发服务退出并由外部拉起，因此这里最多只能写“可控重启”，不能写“零停机”。
- 首版不应承诺“无损回滚”。一旦进入多房间模型，旧版单房间代码无法自然表达多个房间的 Boss 进度；如果业务必须保留这项能力，需要额外做双写，不纳入首版。

## 二、目标与范围

### 2.1 首版目标

- 预置固定房间 ID，玩家可切换当前房间
- 每个房间拥有独立的当前 Boss、血量、伤害榜和击杀奖励
- 切房受冷却限制
- 旧客户端不升级实时协议也能落到默认房间继续玩

### 2.2 首版不做

- 不把 `totalVotes` 和点击总榜拆成按房间统计；它们继续全服共享
- 不做配置热生效；Consul 发布后按现有机制重启服务生效
- 不做分线期间 Boss 进度的无损回滚
- 不做自动删房迁移；房间缩容只做配置回退与人工清理

## 三、当前架构约束

结合当前仓库代码，分线不是单点改动，至少有这些现实约束：

- `backend/internal/core/store.go` 中 `GetSnapshot`、`GetUserState`、`GetState`、`GetBossResources`、`ClickBossPart` 都围绕单一 `boss:current` 运行。
- `backend/internal/events/cache.go` 只有一份公共快照缓存、一份 Boss 资源缓存，用户缓存也只按 `nickname` 建索引。
- `backend/internal/events/hub.go` 的订阅没有房间维度，SSE 初始快照同样直接读取全局状态。
- `backend/internal/httpapi/realtime_socket.go` 的 `snapshot` 和点击后推送都走全局 `StateView`。
- `backend/internal/httpapi/battle_routes.go` 的 `/api/battle/state` 也是全局读取。
- `backend/internal/config/config.go` 的 Consul watcher 在配置变化时会主动退出进程。

因此，本方案必须把房间维度推进到“状态读取入口”，不能只停留在广播过滤。

## 四、修订后的方案

### 4.1 房间真值

房间归属以后端状态为准，不以后端被动相信客户端传参为准。

新增 Redis 键：

```text
{ns}player:room:{nickname}        -> 玩家当前房间
{ns}player:room:cd:{nickname}     -> 切房冷却到期时间戳
{ns}boss:current:{roomID}         -> 房间当前 Boss
```

约束：

- 玩家有昵称时，服务端先按昵称解析当前房间，再读取 Boss 相关状态
- 没有房间记录时，惰性落到 `default_room`
- 房间配置缩容时，如果玩家当前房间已不在配置中，读取时回落到 `default_room`

这样做的好处是：

- 旧客户端仍然可用，因为它不需要新增 `roomID` 协议字段
- 切房冷却只在 `POST /api/rooms/join` 一处判断，读链路不需要重复防御
- SSE、WebSocket、HTTP 首屏状态都能共享同一套房间解析逻辑

### 4.2 Boss 隔离模型

首版只把 Boss 战斗相关状态按房间拆开：

- 当前 Boss：按 `boss:current:{roomID}` 隔离
- Boss 血量、状态、部位、击杀奖励：跟随当前房间 Boss
- Boss 伤害榜：继续沿用 `boss:{bossID}:damage`

这里保留 `boss:{bossID}:damage` 的原因是当前伤害榜本来就按 `bossID` 隔离，只要不同房间激活的是不同 Boss 实例 ID，就天然互不干扰，不必额外再把 `roomID` 拼进伤害榜 key。

但这里有一个前提必须写清楚：

- `bossID` 必须保持全局唯一，不能只做到“单房间内唯一”
- 后台手工激活 Boss 时也不能允许不同房间复用相同 `bossID`
- 原因不只在 Redis 伤害榜；当前 Mongo `boss_history` 也按 `boss_id` 建唯一索引，因此一旦跨房间复用 `bossID`，会同时打坏伤害榜隔离和历史归档唯一性

全局共享不变：

- `totalVotes`
- 点击总榜
- 公告版本
- Boss 模板池 `boss:pool:*`

为避免后台、归档和日志失去上下文，建议把 `roomID` 挂到 `Boss` 结构体上。

但这还不够，必须补一条实现约束：

- 当前 `boss_history` 和 `domain_events` 不是把 Go 结构体原样自动写入 Mongo，而是手工做字段映射
- 因此不能只在 `Boss` 或 `DomainEvent` 结构体上新增 `roomID`，还必须同步修改 Mongo 文档结构、写入映射、读取映射和相关索引
- 如果后台审计日志也要求保留房间上下文，`admin_audit_logs` 也需要单独补 `room_id` 字段与索引；它不会因为 Boss 结构体新增字段而自动获得房间信息

### 4.3 读取链路必须前置房间

所有依赖“当前 Boss”的读取，都必须在进入 Store 时就确定房间。

至少包括：

- `/api/battle/state`
- `GET /api/events` 的初始公共态和个人态
- `/api/ws` 首次 `snapshot`
- `GetBossResources`
- `GetUserState` 中的 `MyBossStats`、天赋战斗态、待消费天赋事件

这部分有两种实现方式：

1. 给相关 Store / StateView 方法显式传 `roomID`
2. 在 Store 内统一通过 `nickname -> roomID` 解析当前房间

首版建议优先采用第二种：

- 对玩家读写链路更简单
- 旧协议改动更少
- 更不容易出现“HTTP 一套房间、SSE 一套房间、WS 一套房间”的分叉

唯一需要显式传 `roomID` 的地方应主要是后台 Boss 管理接口，因为后台操作不一定绑定玩家昵称。

### 4.4 实时链路改造

实时层需要从“全局广播”调整为“按房间广播公共态，按昵称+房间广播个人态”。

建议调整为：

```text
Subscribe(nickname, roomID)
BroadcastRoom(roomID, snapshot)
BroadcastUser(nickname, roomID, userState)
```

补充约束：

- `subscriber` 增加 `roomID`
- `Cache` 的公共快照和 Boss 资源缓存改为按 `roomID` 分桶
- 用户缓存不能再只按 `nickname` 建索引；至少要按 `roomID + nickname` 组合缓存，或在切房时显式失效

`online_count` 不建议在首版做成按房间的实时事件协议变更。更简单的做法是：

- 继续保留现有全局在线人数事件
- 房间在线人数通过 `GET /api/rooms` 返回

这样可以减少前端实时协议改造面。

### 4.5 状态变更分发

`StateChange` 可以增加可选 `RoomID`，但只在“确实只影响单个房间”的场景下使用，例如：

- Boss 被点击
- Boss 被激活 / 停用
- 房间内 Boss 被击杀

对于仍然全局共享、但会出现在房间快照里的内容，例如：

- 公告版本变化
- 点击总榜变化

分发器不应偷懒只发一个全局快照，而应对活跃房间逐个刷新快照并广播。原因很简单：快照里虽然混有全局数据，但也同时带有房间 Boss。

### 4.6 配置模型

配置可以保留在 Consul YAML 中，建议结构如下：

```yaml
room:
  enabled: true
  ids: ["1", "2", "3"]
  default_room: "1"
  switch_cooldown_seconds: 300
```

约束说明：

- `enabled=false` 时仍走当前单房间模式，用于灰度部署
- `ids` 变更需要按现有机制重启服务后生效
- 新增房间后默认没有活跃 Boss，需要后台单独激活
- 删除房间不做自动数据迁移，只做“停止分配 + 人工清理”

### 4.7 API 设计

首版尽量少改现有协议，建议新增两组房间接口，保留现有战斗与实时接口路径不变。

新增：

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/rooms` | GET | 返回配置房间列表、当前房间、剩余冷却、各房间在线人数 |
| `/api/rooms/join` | POST | 切换当前房间，服务端校验目标房间与冷却 |

保持不变但改为房间感知：

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/battle/state` | GET | 按玩家当前房间返回 Boss 相关状态 |
| `/api/events` | SSE | 服务端按玩家当前房间推送对应公共态 |
| `/api/ws` | WS | `hello` 不强制新增 `roomID`，服务端按玩家当前房间绑定订阅 |

后台 Boss 管理接口需要补 `roomID` 维度，但首版不必新开一套管理路径，优先在现有后台 Boss 接口的请求体或查询参数中补充。

## 五、迁移与上线策略

### 5.1 迁移原则

- 不单独写一次性迁移脚本
- 通过 `room.enabled` 做两阶段上线
- 允许默认房间惰性初始化
- 不承诺分线期间的无损回滚

### 5.2 首次启用时的数据兼容

Boss 迁移只做一件事：

- 若 `boss:current:{default_room}` 不存在，且旧 `boss:current` 存在，则复制一份到默认房间

玩家房间只做惰性初始化：

- 读取玩家房间时，若没有 `player:room:{nickname}`，则写入 `default_room`

Mongo 冷数据兼容按“可选字段前向兼容”处理：

- 不要求一次性回填历史 `boss_history`、`domain_events`、`admin_audit_logs` 文档
- 新增的 `room_id` 允许为空，旧文档缺失该字段不应阻塞上线
- 后台或报表如果后续增加按房间筛选，应明确：只有分线上线后新写入、且显式带 `room_id` 的文档才能被准确按房间过滤
- 分线前产生的旧文档应按“单房间历史”或“分线前全服历史”理解，不要在数据层假装它天然属于某个新房间

旧客户端兼容策略：

- 旧客户端继续访问 `/api/battle/state`、`/api/events`、`/api/ws`
- 服务端按当前玩家房间返回数据
- 没有房间记录时自动进入默认房间

### 5.3 上线步骤

1. 部署支持房间能力的新版本代码，但先保持 `room.enabled=false`
2. 验证旧行为未变化
3. 更新 Consul `room` 配置并触发按现有机制重启
4. 启动时执行默认房间 Boss 兼容复制
5. 验证多房间 Boss 独立、切房冷却、旧客户端默认房间兼容

### 5.4 回滚约束

当前最小实现只支持“代码和配置回退”，不支持“分线期间 Boss 进度无损映射回旧单房间模型”。

也就是说：

- 可以回退版本
- 可以关闭 `room.enabled`
- 但分线启用后新增的多房间 Boss 进度，旧版代码不会自动理解

如果业务要求回滚后仍保留默认房间 Boss 进度，需要额外设计默认房间双写旧 `boss:current`，那是第二阶段能力，不纳入本次方案。

## 七、主要风险

| 风险 | 说明 | 缓解方式 |
|------|------|----------|
| 只改广播层，没改读取层 | 首屏和增量状态会看到不同 Boss | 明确把房间维度推进到 Store / StateView 入口 |
| 用户缓存仍只按昵称建索引 | 切房后会拿到旧房间个人态 | 用户缓存改为 `roomID + nickname` 或切房时主动失效 |
| 只给 Go 结构体加 `roomID`，没同步 Mongo 映射 | 冷数据归档、事件日志、后台审计仍然丢失房间上下文 | 显式修改 `mongostore` 的文档结构、索引和读写映射 |
| `bossID` 不是全局唯一 | Redis 伤害榜和 Mongo `boss_history` 唯一索引会互相冲突 | 约束所有房间使用全局唯一 Boss 实例 ID，后台手工激活也不能复用 |
| 宣称零停机 | 与当前 Consul 配置重启机制不符 | 文档改为“可控重启” |
| 宣称无损回滚 | 旧版单房间模型无法表达多房间 Boss | 首版不承诺，必要时另做默认房间双写 |
| 房间缩容 | 现有玩家和 Boss 数据无自动迁移 | 首版只做回落默认房间与人工清理 |

## 八、验收标准

实现时至少要满足这些可验证目标：

1. 同一个玩家在房间 1 和房间 2 看到的 `boss.id`、`currentHp`、伤害榜彼此独立。
2. 房间 1 的 Boss 点击不会改动房间 2 的 Boss 血量和房间 2 的 Boss 伤害榜。
3. `totalVotes` 和点击总榜仍保持全服共享，不因房间切换而被拆分。
4. `/api/battle/state`、`/api/events` 初始态、`/api/ws` 首次 `snapshot` 对同一玩家返回同一房间视角。
5. 切房接口会校验房间是否合法、冷却是否到期，并在成功后影响后续读取与实时推送。
6. 不升级客户端协议时，旧客户端仍能进入默认房间并正常打 Boss。
7. 配置变更后的服务重启不会导致房间状态错乱，最多只产生已有架构允许的短时重连。

## 九、实施顺序

1. 先改配置与 Store 的房间真值解析，确保读写入口统一。
2. 再改 `/api/battle/state`、SSE、WebSocket 的首屏读取，让首屏状态先正确。
3. 再改 `Hub / Cache / Dispatcher`，把增量推送切到按房间广播。
4. 再补房间列表、切房接口和后台 Boss 房间维度。
5. 最后接前端房间选择 UI，并补齐回归测试。
