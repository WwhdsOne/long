# 房间开放时间规则与责任链设计

> 状态：已审核，待实现
> 日期：2026-05-07
> 结论：采用“周计划 OR 临时窗口”的最小开放模型；未配置时间规则的房间维持现状；后端用责任链拆分时间开放判定；活动房关闭后禁止新进入，玩家读取当前房间时统一回退 `hall`，并且活动房全量禁用挂机。

## 一、背景与目标

当前房间是否可进入，主要由“有没有活动 Boss”与“循环是否开启”决定，缺少稳定的时间开放规则。这会带来两个直接问题：

- 活动房只能靠人工开关，不能表达“每周固定时段开放”
- 临时加开活动时，需要直接改业务开关，容易污染原有房间可加入语义

本次设计目标：

- 为房间增加两层时间开放规则：
  - 每周固定计划
  - 临时日期窗口
- 生效口径固定为并集：
  - `周计划命中 || 临时窗口命中`
- 临时窗口只负责“额外开放”，不承担“临时关闭”语义
- 把“业务上可加入”和“时间上可开放”拆开，避免继续堆进单个 `roomJoinable`
- 用责任链组织开放判定，给后续“维护锁”“临时封禁”“白名单开放”等规则留插槽

## 二、范围与非目标

### 2.1 本次要做

- 新增房间时间规则存储模型
- 新增按房间的开放判定责任链
- 扩展后台房间管理接口与页面，支持维护周计划和临时窗口
- 扩展玩家房间列表返回字段，明确当前是否开放与状态文案
- 活动房关闭后阻止新进入，并在读取当前房间时回退到 `hall`
- 配置过任意时间规则的房间，统一视为活动房，并禁止挂机

### 2.2 本次不做

- 不支持“临时关闭覆盖周计划”
- 不支持跨午夜的单条周计划时间段
- 不新增“维护锁”“封禁窗口”“白名单开放”具体功能
- 不做主动定时扫描踢人
- 不做活动房 AFK 迁移与补偿

## 三、核心规则

### 3.1 时间规则模型

新增 `RoomSchedule`：

```go
type RoomSchedule struct {
    WeeklyWindows map[Weekday][]DailyTimeRange `json:"weeklyWindows"`
    AdHocWindows  []AbsoluteTimeRange          `json:"adHocWindows"`
}
```

新增类型：

```go
type DailyTimeRange struct {
    StartSeconds int `json:"startSeconds"`
    EndSeconds   int `json:"endSeconds"`
}

type AbsoluteTimeRange struct {
    StartAt int64 `json:"startAt"`
    EndAt   int64 `json:"endAt"`
}
```

约束：

- 时区固定为北京时间
- 周计划支持每天多段
- 周计划单段不允许跨午夜，必须满足 `startSeconds < endSeconds`
- 临时窗口使用绝对时间戳，允许自然跨天
- 未配置任何时间规则的房间，视为“无时间限制”，保持现有行为

### 3.2 生效口径

房间时间开放口径固定为：

```text
weekly_hit || ad_hoc_hit
```

解释：

- 命中周计划：开放
- 未命中周计划，但命中临时窗口：开放
- 两者都未命中：关闭
- 临时窗口不会覆盖、关闭、否定周计划，只负责额外加开

## 四、责任链判定器

### 4.1 设计目标

不把时间开放继续写成一坨 `if else`，而是拆成独立责任节点。这样后续新增高优先级拒绝规则时，只需要往链中插入新节点，不需要重写主判断流程。

### 4.2 建议结构

```go
type RoomOpenCheckContext struct {
    Now           time.Time
    RoomID        string
    Schedule      RoomSchedule
    HasActiveBoss bool
    CycleEnabled  bool
}

type RoomOpenDecision struct {
    Allowed     bool
    Reason      string
    MatchedRule string
}

type RoomOpenChecker interface {
    Check(ctx context.Context, in RoomOpenCheckContext) (RoomOpenDecision, bool, error)
}
```

其中第二个返回值 `handled` 表示：

- `true`：当前节点已给出最终结论，责任链停止
- `false`：当前节点不做最终裁决，继续后续节点

### 4.3 建议责任链顺序

#### 1. `RoomScheduleBypassChecker`

- 条件：该房间完全没有配置任何时间规则
- 结论：直接返回 `Allowed=true`
- 原因：兼容历史房间，不让未配置房间被默认关掉

#### 2. `RoomWeeklyScheduleChecker`

- 判断当前北京时间是否命中本周当天任一时间段
- 命中时返回：
  - `Allowed=true`
  - `MatchedRule=weekly`
- 未命中则继续后续节点

#### 3. `RoomAdHocWindowChecker`

- 判断当前时间是否命中任一临时窗口
- 命中时返回：
  - `Allowed=true`
  - `MatchedRule=ad_hoc`
- 未命中则继续后续节点

#### 4. `RoomScheduleDenyChecker`

- 条件：前面都未命中，且该房间存在时间规则
- 结论：
  - `Allowed=false`
  - `Reason=out_of_schedule`

### 4.4 责任链收益

- 规则增删不污染主流程
- 以后加“临时封禁窗口”时，只需插入高优先级 `DenyChecker`
- 以后加“管理员维护锁”“特殊白名单”时，也只需新增节点
- 单测可按节点逐个覆盖，不必每次跑整条房间链路

## 五、业务可加入与时间开放分层

当前 `roomJoinable` 混合了业务开放和时间开放语义，建议拆成三层：

- `roomBusinessJoinable`
  - 当前 Boss 存在且 active，或循环开启
- `roomScheduleOpenNow`
  - 通过责任链判断当前是否命中时间规则
- `roomJoinable`
  - `roomBusinessJoinable && roomScheduleOpenNow`

这样拆分后的收益：

- 后台、房间列表、切房接口可以分别展示“业务上可进”和“当前时间开放”
- 即使房间时间窗打开，但当前没有 Boss、也没开循环，也不会误导成可加入
- 后续房间关闭原因可以更明确区分为：
  - 业务未开启
  - 时间未开放

## 六、存储与接口

### 6.1 Redis 存储

按房间新增 schedule 键，值直接存 JSON，避免周计划多段拆成难维护的 hash 字段。

建议键：

```text
{ns}room:schedule:{roomID}
```

原因：

- 周计划与临时窗口都是层级结构，JSON 更适合整体读写
- 后台编辑时天然就是全量提交与覆盖
- 后续扩字段时兼容成本更低

### 6.2 后台接口

扩展后台房间管理：

- `GET /api/admin/rooms`
  - 返回房间显示名
  - 返回当前开放状态
  - 返回周计划
  - 返回临时窗口

- `PUT /api/admin/rooms/:roomId`
  - 请求体包含：
    - `displayName`
    - `weeklyWindows`
    - `adHocWindows`

后端校验：

- `roomId` 必须合法
- 周计划时间段必须满足：
  - 时间格式合法
  - `startSeconds < endSeconds`
- 临时窗口必须满足：
  - `startAt < endAt`

### 6.3 玩家接口

扩展 `GET /api/rooms` 返回字段：

- `joinable`
- `openNow`
- `openStatusText`

建议文案口径：

- `常规开放中`
- `临时开放中`
- `当前未开放`

扩展 `POST /api/rooms/join` 行为：

- 若房间当前不在开放时间窗内，返回 `ROOM_NOT_JOINABLE`

## 七、房间关闭与挂机规则

### 7.1 房间关闭后的玩家处理

本次不依赖定时扫描主动踢人。

统一策略：

- 房间关闭后，禁止新进入
- 玩家读取当前房间时，如果房间已不开放，服务端直接回退到 `hall`

这样可以保证以下口径一致：

- SSE
- 房间列表
- 战斗页状态读取

### 7.2 挂机规则

只要房间配置了任意时间规则，就视为活动房。

活动房统一：

- 不允许启动 AFK
- AFK 玩家不能切入活动房

这样处理后：

- 活动房关闭时，无需再处理复杂挂机迁移
- AFK 结算模型不会和活动房时间规则相互打架

## 八、前端与后台交互设计

### 8.1 后台房间管理页

新增两块编辑区：

- 每周计划
  - 周一到周日分别维护
  - 每天支持多段 `HH:mm-HH:mm`
- 临时窗口
  - 支持多条绝对日期时间范围

输入方式建议：

- 周计划使用时间选择器
- 临时窗口使用日期时间选择器

提交转换规则：

- 周计划转日内秒数
- 临时窗口转 Unix 时间戳

### 8.2 玩家房间列表

房间未开放时：

- 按钮禁用
- 展示开放状态文案

配置了时间规则的房间额外展示：

- `该房间不支持挂机`

## 九、测试计划

### 9.1 责任链单测

- 未配置时间规则时直接允许
- 命中周计划时允许
- 命中临时窗口时允许
- 周计划和临时窗口都未命中时拒绝
- 周计划与临时窗口重叠时仍为允许

### 9.2 核心房间单测

- `roomBusinessJoinable` 为 false 时，即使时间命中也不可加入
- `roomBusinessJoinable` 为 true 且时间命中时可加入
- 房间关闭后 `ResolvePlayerRoom` 回退 `hall`

### 9.3 路由测试

- 后台保存多段周计划成功
- 后台保存多条临时窗口成功
- 非法时间段被拒绝
- 关闭房间 join 返回 `ROOM_NOT_JOINABLE`

### 9.4 前端静态测试

- 后台页存在每周计划和临时窗口编辑区
- 房间入口根据 `openNow` / `joinable` 禁用
- 活动房出现“禁止挂机”提示

## 十、默认假设

- 开放规则固定为：`周计划 OR 临时窗口`
- 临时窗口只做额外开放，不做封禁
- 周计划支持每天多段
- 临时窗口支持多条
- 时区固定为北京时间
- 不配置时间规则的房间维持现状
- 责任链首版只负责“时间开放判定”

## 十一、落地建议

建议按以下顺序推进：

1. 先落后端时间模型、Redis 存储和责任链判定器
2. 再拆分 `roomBusinessJoinable` / `roomScheduleOpenNow` / `roomJoinable`
3. 然后补后台接口与房间管理页
4. 最后补玩家房间列表展示、join 拒绝与 AFK 限制

这样可以先把最容易出错的开放判定核心单测补齐，再逐步接入页面与交互层。
