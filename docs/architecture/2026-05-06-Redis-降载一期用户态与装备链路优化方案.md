# Redis 降载一期：用户态与装备链路低风险优化方案

> 状态：待 review
> 日期：2026-05-06
> 结论：本轮只做用户态读取链路减载，不改 Redis key 结构、不改前端协议、不改 Boss 与 AFK 主写路径，优先压低 `GetUserState`、装备实例读取和装备定义读取带来的 Redis 命令数与网络流量。

## 一、背景

当前 Redis 热点里，用户态和装备链路已经成为一条独立的高频读路径：

- HTTP 首屏、登录后资料页、SSE/WebSocket 首次同步都会走 `GetUserState`
- 实时事件分发里的个人态广播也复用 `GetUserState`
- 装备栏、背包、战斗属性都依赖装备实例和装备定义读取

现状下，这条链路的问题不是某一个命令特别慢，而是：

- 单次用户态构建串读分支过多
- 同一请求内存在重复扫 `player-instances:*`
- 同一请求内会重复读取相同 `equip:def:*`
- 增量用户广播虽然最终不发 `Loadout` / `CombatStats`，但分发前仍然做了全量读取

因此本轮目标不是“重构用户态模型”，而是先做一轮边界清晰、风险低、能直接降低 Redis 压力的代码路径收敛。

## 二、当前实现现状

### 2.1 `GetUserState` 当前是全量串读

当前实现位于：

- `backend/internal/core/store.go`

`GetUserState(ctx, nickname)` 在一个请求里会依次读取：

- 房间信息
- 资源与 Boss 击杀数
- `UserStats`
- `Loadout`
- `Inventory`
- `CombatStats`
- `RecentRewards`
- `Tasks`
- 战斗点击皮肤状态
- Boss 个人态
- 天赋事件与天赋战斗态

这意味着只要调用方想拿一个“实时个人态”，就会顺带触发多条本不总是必要的 Redis 读取。

### 2.2 装备实例当前会在同一请求里重复读取

相关函数位于：

- `backend/internal/core/store.go`

当前调用关系是：

1. `GetUserState`
2. `loadoutForNickname`
3. `itemInstancesByIDForNickname`
4. `inventoryForNickname`
5. `itemInstancesByIDForNickname`

也就是说，同一次 `GetUserState` 里：

- `SMEMBERS player-instances:{nickname}` 会做两次
- `Pipeline HMGET instance:{id}` 会做两轮

如果用户背包实例较多，这部分会直接放大 Redis 请求数和返回体积。

### 2.3 装备定义读取缺少请求内去重和本地缓存

当前 `loadoutForNickname` 和 `inventoryForNickname` 都会调用 `getEquipmentDefinition(ctx, itemID)`。

现状问题有两层：

- 同一请求里，多个实例只要 `itemID` 相同，仍可能重复读同一个 `equip:def:{itemID}`
- 跨请求没有本地缓存，同一件常见装备会被反复回源 Redis

这类数据本质上是展示型元数据，变化频率远低于用户态读取频率。

### 2.4 分发器当前没有“轻量用户态”路径

相关函数位于：

- `backend/internal/events/dispatcher.go`
- `backend/internal/events/hub.go`

当前分发器行为是：

- 先根据变更类型决定 `includeProfile`
- 再统一 `RefreshUsers`
- 最后 `BroadcastUser(nickname, userState, includeProfile)`

而 `includeProfile=false` 只影响最终序列化时是否带：

- `Loadout`
- `CombatStats`

但不会阻止前面的 `GetUserState` 去读取：

- `Inventory`
- `Tasks`
- `Loadout`
- `CombatStats`

这导致“已经瘦身的 payload”前面仍然有一整套重读成本。

## 三、方案目标

本轮只追求四个结果：

1. 增量 `user_state` 广播不再触发背包、装备栏、任务等非必要读取
2. 单次全量用户态构建中，装备实例集合只读取一次
3. 单次用户态构建中，相同 `itemID` 的装备定义只解析一次
4. 跨请求对 `equip:def:*` 增加短 TTL 本地缓存，减少重复回源 Redis

明确不做：

- 不改 Redis key 结构
- 不改前端协议字段与事件名
- 不改 AFK 扫描逻辑
- 不改 Boss 战斗写链路
- 不引入跨进程缓存一致性设计

## 四、推荐方案

### 4.1 拆分“全量用户态”和“实时轻量用户态”

保留现有：

- `GetUserState(ctx, nickname)`

它继续服务：

- HTTP 全量资料读取
- SSE/WebSocket 首次同步
- 需要完整 profile 的用户态广播

同时新增一个仅供内部使用的轻量入口，例如：

- `getRealtimeUserState`
- 或等价命名的内部函数

轻量态只读取当前增量 `user_state` 真正会发送的字段：

- `UserStats`
- `MyBossStats`
- `MyBossKills`
- `TotalBossKills`
- `RoomID`
- `Gold`
- `Stones`
- `TalentPoints`
- `RecentRewards`
- `TalentEvents`
- `TalentCombatState`
- `EquippedBattleClickSkinID`
- `EquippedBattleClickCursorImagePath`

其中有一个语义必须保持不变：

- `TalentEvents` 继续沿用当前“读取即消费”的一次性事件语义
- 轻量态仍然读取 `TalentCombatState`
- 轻量态仍然消费 pending talent events 队列，不能改成只读、延后消费或跳过消费

轻量态明确不读取：

- `Inventory`
- `Tasks`
- `Loadout`
- `CombatStats`

这样可以把“增量广播瘦身”从“只瘦 payload”推进到“连读取也瘦下来”。

### 4.2 分发器按场景切换读取路径

当前 `includeProfile` 的判断已经存在，本轮直接复用：

- `includeProfile=false`
  - 走轻量用户态读取
- `includeProfile=true`
  - 继续走完整 `GetUserState`

这意味着：

- 大多数实时个人态广播都会走轻量路径
- 只有 `equipment_changed`、`equipment_meta_changed` 这类明确需要资料真值刷新的场景，才继续走全量用户态

同时要求：

- `includeProfile=false` 只裁掉 `Loadout` / `CombatStats` 相关重读
- 不改变 `TalentEvents` 的一次性消费语义

方案边界清晰，且与当前 `hub.go` 的序列化策略一致。

### 4.3 合并装备实例读取，单次组装后复用

在一次完整用户态构建中，新增内部装备快照组装步骤，统一完成：

1. `SMEMBERS player-instances:{nickname}`
2. `Pipeline HMGET instance:{id} ...`

然后把读取结果同时供下面三类逻辑复用：

- `loadout`
- `inventory`
- `combatStats`

具体调整建议：

- `loadoutForNickname` 不再自己触发一次实例集合读取
- `inventoryForNickname` 不再重新触发第二次实例集合读取
- `combatStatsForNickname` 继续消费最终 `loadout`，不额外回头扫实例

这轮的重点不是抽象层次，而是确保一次 `GetUserState` 内部只做一轮实例集合读取。

### 4.4 `loadout:{nickname}` 从 `HGETALL` 收窄为固定槽位 `HMGET`

当前装备栏槽位是固定集合，本轮建议把：

- `HGETALL loadout:{nickname}`

改成固定槽位读取：

- `weapon`
- `helmet`
- `chest`
- `gloves`
- `legs`
- `accessory`

已确认当前实际数据不存在 `armor` 历史槽位，本轮按规范 6 槽位直接收窄读取，不再为 `armor` 预留单独兼容路径；实现时可同步删除对应历史兼容测试。

收益是：

- 避免把历史脏字段、未知字段、无关字段整哈希拉回
- 保持现有 6 槽位业务语义不变

由于槽位集合当前是静态的，这属于低风险收窄读取。

### 4.5 装备定义增加“请求内去重 + 进程内短 TTL 缓存”

本轮建议分两层减少 `equip:def:*` 读取。

第一层：请求内去重

- 在单次用户态构建内，先从所有实例收集唯一 `itemID`
- 对唯一 `itemID` 读取装备定义
- 后续构造 `loadout`、`inventory` 时复用已解析结果

第二层：进程内短 TTL 缓存

- 缓存位置：`Store` 内部
- key：`itemID`
- value：已解析 `EquipmentDefinition`
- TTL：`30s`

读取顺序：

1. 先查本地缓存
2. 未命中再读 Redis
3. 读到 Redis 结果后回填缓存

缓存失效规则：

- `Store.SaveEquipmentDefinition` 成功写入后，显式删除对应缓存项
- `Store.DeleteEquipmentDefinition` 成功删除后，显式删除对应缓存项
- 不把缓存失效只挂在路由层，避免后续新增调用入口时漏失效

本轮接受的代价是：

- 跨进程不一致不处理
- 允许最多 30 秒展示型元数据延迟

对这类后台编辑频率很低、前台读取频率很高的元数据，这个权衡是合理的。

## 五、实现范围

### 5.1 主要改动文件

- `backend/internal/core/store.go`
- `backend/internal/core/admin.go`
- `backend/internal/events/cache.go`
- `backend/internal/events/dispatcher.go`
- 相关测试文件

其中：

- `store.go` 负责全量态/轻量态拆分、装备快照复用、定义缓存
- `admin.go` 负责装备定义写入口上的缓存失效联动
- `events/cache.go` 负责为分发器暴露轻量读取入口
- `dispatcher.go` 负责按 `includeProfile` 选择读取路径

### 5.2 明确不改的文件和链路

- 不改前端
- 不改 Redis migration
- 不改 Boss 点击主写路径
- 不改 AFK 定时处理
- 不改 SSE/WebSocket 事件名与对外字段

## 六、测试与验收

### 6.1 store 层测试

至少补下面几类测试：

1. 全量用户态与轻量用户态在共享字段上的值一致
2. 轻量用户态不触发 `inventory` / `loadout` / `task` 相关读取分支
3. 单次全量用户态构建中，装备实例集合只读取一次
4. 同一请求内多个相同 `itemID` 仅解析一次装备定义
5. 装备定义缓存命中后，重复读取不再回源 Redis
6. 装备定义修改或删除后，下一次读取不返回旧缓存值
7. 轻量用户态第一次读取能返回 pending `TalentEvents`，第二次读取不会重复返回

### 6.2 events 层测试

至少补下面两类测试：

1. `includeProfile=false` 时，分发器走轻量用户态路径
2. `includeProfile=true` 时，分发器仍走全量用户态路径
3. `includeProfile=false` 时，广播仍带当前应有的 `TalentEvents` / `TalentCombatState`

### 6.3 验收指标

本轮验收不以“彻底解决所有 Redis 热点”为标准，而以“用户态读链路明显收窄”为标准。

重点观察：

- 增量 `user_state` 广播不再触发背包/装备链路读取
- 单次全量用户态构建的 Redis 命令数明显下降
- `HGETALL loadout:*` 在命令统计里消失
- `SMEMBERS player-instances:*` 总量下降
- `equip:def:*` 的命令数和网络流量占比下降

建议重点盯的线上指标：

- `HMGET` 总量
- `SMEMBERS` 总量
- `HGETALL loadout:*` 是否清零
- `equip:def:*` 相关流量与命中次数

## 七、风险与边界

### 7.1 轻量态和全量态可能出现实现分叉

风险点在于：

- 两套读取路径如果长期演进不一致，可能导致字段真值漂移
- 轻量态如果漏掉 pending `TalentEvents` 消费，可能出现事件漏发或重复发

控制方式：

- 轻量态只覆盖当前已明确在增量 payload 中使用的字段
- 对共享字段补“全量态与轻量态一致性”测试
- 对 `TalentEvents` 补“一次读取后即消费完成”的语义测试
- 保持轻量态为内部函数，不对外暴露新协议

### 7.2 本地缓存存在短时陈旧窗口

本轮接受：

- 单进程内最多 30 秒装备定义缓存陈旧

前提是：

- 这类数据属于展示型元数据
- 管理端编辑频率远低于前台读取频率

如果后续后台频繁改装备定义，再评估是否需要二期做版本号或 Pub/Sub 失效。

### 7.3 本轮不解决 Boss 与 AFK 热点

这份方案故意不把范围扩散到：

- AFK 全量扫描
- Boss `TxPipeline`
- `ZINCRBY` 慢查询

原因很简单：

- 这些路径的收益点和风险模型不同
- 与用户态读取链路不属于同一类问题
- 混在一轮实施里会显著放大 review 和回归成本

## 八、推荐实施顺序

建议分两步走：

### 第一步：先做最确定的减载

1. 分发器切轻量用户态
2. 装备实例单次读取复用
3. `loadout:{nickname}` 从 `HGETALL` 改 `HMGET`

### 第二步：再加装备定义缓存

1. 请求内 `itemID` 去重
2. `Store` 内部短 TTL 缓存
3. 管理端变更触发缓存失效

这样做的好处是：

- 先拿到读取路径收窄的确定收益
- 再引入带状态的缓存逻辑，便于分段验证和回滚

## 九、Review 重点

这份方案在 review 时，建议重点看四个问题：

1. 轻量用户态字段集合是否刚好覆盖当前增量 `user_state` 真实需要的字段
2. 轻量路径是否完整保留 `TalentEvents` 的一次性消费语义
3. `loadout` 固定 6 槽位 `HMGET` 是否与当前装备系统约束一致
4. 装备定义 `30s` 本地缓存窗口是否可接受
5. 装备定义新增、编辑、删除时的缓存失效是否都收敛在底层写入口

## 十、结论

这轮 Redis 降载不追求大改模型，而是优先修三类明显的重复读取：

- 增量个人态仍走全量构建
- 同一请求内重复扫装备实例
- 同一请求内和跨请求重复读装备定义

如果 review 通过，本轮实施后应能较稳定地看到：

- 增量 `user_state` 读取链路变轻
- 全量用户态装备相关命令数下降
- `equip:def:*` 与 `player-instances:*` 在 Redis 统计中的占比下降
