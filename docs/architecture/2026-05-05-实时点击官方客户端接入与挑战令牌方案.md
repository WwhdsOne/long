# 实时点击官方客户端接入与挑战令牌方案

> 状态：待实施
> 日期：2026-05-05
> 结论：不再继续堆叠时间段限流；实时点击链路改为“玩家登录会话 + 页面挑战令牌 + WebSocket 握手来源校验”三段接入门槛，只保护 `WebSocket click` 主链路，不影响现有离页挂机。

## 一、背景

当前实时点击链路已经迁移到：

- 前端通过 `WebSocket /api/ws` 发送点击
- 服务端在 `executeButtonClick` 前执行 `ClickGuard`
- 公共态和个人态继续走 `SSE /api/events`、`/api/events/me` 下行推送

现状问题不是“完全没有限流”，而是：

- 现有限流主要是频率门槛
- 随机间隔脚本可以压在阈值下长期运行
- 仅靠 `IP` / `nickname` 的滑窗计数，无法区分“官方页面发起的点击”和“裸脚本直连”

本次目标不是做完整行为风控，而是先把实时点击入口收紧到：

- 必须先有玩家登录会话
- 必须先从官方页面拿到短期挑战令牌
- `WebSocket` 握手必须通过基本来源校验

## 二、已确认的架构事实

- 点击主链路在 `backend/internal/httpapi/realtime_socket.go`。
- 真正触发限流的入口在 `backend/internal/httpapi/realtime_helpers.go` 的 `executeButtonClick(...)`。
- 当前 `WebSocket` 升级器 `CheckOrigin` 直接放行。
- 前端实时连接在 `frontend/src/utils/realtimeTransport.js`，由页面先建 `WebSocket`，再发 `hello`，后续发二进制 `click`。
- 仓库里存在 `/api/auto-click/*` 兼容接口，但当前未注入实际控制器。
- 真正在跑的挂机是 `AfkService -> AttackBossPartAFK*`，不经过 `ClickGuard`，也不走实时点击主链路。

因此，本方案只改：

- `页面 -> /api/ws/challenge -> /api/ws -> click`

不改：

- `AfkService`
- `AttackBossPartAFK*`
- 后台管理接口

## 三、目标与非目标

### 3.1 目标

- 只允许“带有效玩家登录态的官方页面会话”发起实时点击。
- 抬高裸 `curl` / `python requests` / `Go-http-client` / 自写 Java 客户端的接入成本。
- 在不改核心战斗逻辑的前提下，收紧 `WebSocket` 握手。
- 尽量少改前端页面状态层，只在实时连接建立前补一个挑战请求。
- 不影响现有挂机链路。

### 3.2 非目标

- 不做行为风控、方差检测、长周期画像。
- 不尝试识别“脚本语言”本身。
- 不承诺拦住 `Selenium` / `Playwright` / `puppeteer` 这类浏览器自动化。
- 不新增 Redis/Mongo 风控模型。
- 不把所有 HTTP 接口都升级成同样的挑战机制。

## 四、方案总览

接入链路改成四步：

1. 玩家先完成现有登录，拿到 `long_player_session`。
2. 官方前端在建立实时连接前，请求 `GET /api/ws/challenge`。
3. 服务端验证玩家登录态后，签发一个短期 `ws challenge token`。
4. 前端把该 token 带到 `WebSocket /api/ws` 握手 URL 上；服务端在升级前校验：
   - 玩家登录态仍然有效
   - challenge token 未过期、内容匹配
   - `Origin / Host` 基本一致
   - `User-Agent` 不为空，且不命中明显脚本 UA 黑名单

只有握手通过的连接，才允许进入现有 `hello / click / sync_request` 消息流。

## 五、挑战令牌设计

### 5.1 新增接口

新增：

- `GET /api/ws/challenge`

请求要求：

- 必须携带有效 `long_player_session` Cookie
- 默认同站请求即可，不要求前端额外传参

响应体建议：

```json
{
  "token": "base64url(payload).base64url(signature)",
  "expiresAt": 1746432310
}
```

### 5.2 令牌负载

负载建议包含：

- `nickname`
- `path`，固定为 `/api/ws`
- `nonce`
- `issuedAt`
- `expiresAt`
- `clientIp`
- `uaHash`

说明：

- `path` 防止该 token 被复用到别的接口。
- `clientIp` 绑定当前来源地址，降低跨端复用价值。
- `uaHash` 绑定当前 UA，降低不同客户端之间互抄 token 的价值。
- `nonce` 用于避免同秒内令牌完全相同，便于后续如果要做一次性消费时扩展。

### 5.3 签名方式

首版不额外引入新配置，直接复用当前玩家 session token 作为签名密钥来源。

建议做法：

- 使用 `HMAC-SHA256`
- `secret = playerSessionCookieValue`
- `message = canonicalJSON(payload)` 或约定好的 `|` 拼接串

这样有两个直接收益：

- 前端 JS 不需要读取 `HttpOnly` Cookie 也能拿到挑战令牌
- 就算 challenge token 泄露，没有对应的玩家 session cookie 也无法独立通过校验

### 5.4 有效期

建议首版：

- `TTL = 10 秒`

原因：

- 足够覆盖页面发起 `fetch('/api/ws/challenge')` 后立即建连
- 过期窗口短，降低抓包复用价值
- 不需要做复杂刷新，只在每次重连前重新取一次 challenge

首版不强制一次性消费，不新增 Redis nonce 表。

## 六、握手校验规则

`/api/ws` 在执行 `upgrader.Upgrade(...)` 之前新增一层接入校验。

### 6.1 强校验

以下任一失败，直接拒绝握手：

- 玩家登录态不存在或已失效
- 缺少 `challenge` 参数
- challenge token 签名错误
- challenge token 已过期
- challenge payload 的 `nickname` 与当前登录玩家不一致
- challenge payload 的 `path` 不是 `/api/ws`
- challenge payload 的 `clientIp` 与当前请求 IP 不一致
- challenge payload 的 `uaHash` 与当前请求 UA 不一致

### 6.2 来源校验

来源校验只做“官方页面应满足的最小条件”，不做过度浏览器指纹化。

建议规则：

- `Origin` 存在时，必须与当前 `scheme + host` 同站
- `Host` 必须存在
- `Origin` 缺失时，不因这一条单独放行未知来源；仍需依赖 challenge token

这里的重点不是“靠 Origin 单独防脚本”，而是：

- challenge token 负责“先走过官方页面请求”
- `Origin / Host` 负责挡掉明显错误的握手来源

### 6.3 明显脚本 UA 拒绝

首版只拦明显低级 UA，避免误伤正常浏览器：

- 空 UA
- 包含 `curl`
- 包含 `wget`
- 包含 `python-requests`
- 包含 `python-urllib`
- 包含 `go-http-client`
- 包含 `java/`
- 包含 `okhttp`
- 包含 `apache-httpclient`

注意：

- 这只是低成本拦截，不是安全边界。
- 只要对方主动伪装浏览器 UA，这条就可能失效。

## 七、前端接入改造

前端只做最小改动。

### 7.1 建连前先拿 challenge

当前 `frontend/src/utils/realtimeTransport.js` 是直接：

1. 构造 `/api/ws`
2. `new WebSocket(url)`
3. 打开后发送 `hello`

改为：

1. `fetch('/api/ws/challenge', { credentials: 'include' })`
2. 解析 `token`
3. 构造 `/api/ws?challenge=...`
4. 再 `new WebSocket(url)`

### 7.2 重连逻辑

每次主动重连或断线重连时，都重新请求 challenge。

不做：

- 本地缓存长效 token
- 多次重连复用旧 token

这样可以保证 challenge 的短期性，也避免因为 token 过期导致重连逻辑变脏。

### 7.3 失败回退

如果 challenge 获取失败：

- 视为实时主链路不可用
- 维持当前降级策略，退回 `SSE`
- 前端错误文案明确区分：
  - “实时鉴权失败，请重新登录后重试”
  - “实时主链路暂时不可用，已切回兼容模式”

## 八、后端改造边界

首版改动边界控制在 `httpapi`。

建议新增或调整：

- `backend/internal/httpapi/realtime_guard.go`
  - challenge payload 定义
  - 签发函数
  - 校验函数
  - `Origin / UA` 检查函数
- `backend/internal/httpapi/realtime_routes.go` 或现有实时路由注册处
  - 注册 `GET /api/ws/challenge`
- `backend/internal/httpapi/realtime_socket.go`
  - 升级前执行 challenge / 来源校验
- `frontend/src/utils/realtimeTransport.js`
  - 建连前先取 challenge
- 对应前后端测试

明确不改：

- `Store.ClickButton`
- `ratelimit.Limiter`
- `AfkService`
- `AttackBossPartAFK*`

## 九、错误码与返回口径

建议新增错误码，避免前端只能看到模糊断连：

### 9.1 `GET /api/ws/challenge`

- `401 UNAUTHORIZED`
  - 玩家未登录或 session 失效
- `503 CHALLENGE_UNAVAILABLE`
  - 服务端生成 challenge 失败

### 9.2 `GET /api/ws`

握手失败建议返回：

- `401 WS_UNAUTHORIZED`
  - 玩家登录态失效
- `401 WS_CHALLENGE_REQUIRED`
  - 缺少 challenge
- `401 WS_CHALLENGE_INVALID`
  - challenge 无效、过期、签名不匹配
- `403 WS_ORIGIN_INVALID`
  - 来源不合法
- `403 WS_CLIENT_REJECTED`
  - 明显脚本 UA

## 十、验证与测试建议

### 10.1 后端

至少补这些测试：

- 未登录请求 `/api/ws/challenge` 返回 `401`
- 已登录请求 `/api/ws/challenge` 返回 token
- challenge 过期时握手失败
- challenge 的 `IP / UA / nickname / path` 不匹配时握手失败
- 明显脚本 UA 握手失败
- 合法 challenge + 合法来源时握手通过
- 现有实时 `hello / click_ack / room_state` 测试继续通过

### 10.2 前端

至少补这些测试：

- 建连前先请求 `/api/ws/challenge`
- 建连 URL 带上 challenge
- challenge 获取失败时回退 `SSE`
- 重连时重新获取 challenge，而不是复用旧值

## 十一、上线策略

建议分两步：

1. 先上线 challenge 接口与校验逻辑，但保留观测开关
2. 前端切到新建连流程后，再把 `/api/ws` 的“无 challenge 放行”彻底关闭

如果不想做开关，至少要保证：

- 前后端同批次发布
- 前端先带 challenge，再要求后端强制 challenge

## 十二、风险与边界

### 12.1 能挡住什么

- 直接拿 `curl` / `python requests` / `Go-http-client` 打 `WebSocket` 的低级脚本
- 没有玩家登录态、只想伪造 `nickname` 的脚本
- 没有先走官方页面 challenge 流程的裸客户端

### 12.2 挡不住什么

- 能完整模拟浏览器请求头和连接流程的客户端
- 能先调 challenge 接口、再建连的高级脚本
- 基于真实浏览器环境的自动化工具

所以这个方案的定位很明确：

- 不是终局安全方案
- 是先把“接入门槛”从纯频率控制提升到“官方页面会话 + 短期握手令牌”
