# 实时公共态广播节流与载荷收敛方案

> 状态：待实施
> 日期：2026-05-05
> 结论：点击主链路保持即时 `click_ack`，公共态广播从“尾沿防抖”改为“固定窗口节流”，并把 `room_state` 与低优先级榜单从高频点击广播里拆出。

## 一、问题定义

当前点击手感与服务器带宽之间出现直接冲突：

- `debounce_ms: 200` 时，公共态更新体感明显发黏
- `debounce_ms: 100` 时，体感改善，但公共广播频率上升
- 小带宽服务器更怕“高频广播 x 广播扇出 x 单包体积”同时放大

因此优化目标不能只盯着 `debounce_ms` 数字本身，而要同时控制：

- 广播频率上限
- 单次消息体大小
- 每次点击触发的广播扇出范围

## 二、当前实现现状

### 2.1 点击后的两段式同步

当前点击链路分成两段：

1. `click_ack`
   - 由 `WebSocket` 立即返回
   - 用于即时更新当前玩家的点击反馈、Boss 掉血、个人资源等
2. `public_state`
   - 由 `events.Dispatcher` 延后广播
   - 用于同步公共数字和公共战斗态

因此当前“卡感”主要不是点击请求本身慢，而是公共态广播被尾沿防抖推迟。

### 2.2 当前公共态消息类型

现在与公共态相关的下行消息一共有 3 类：

#### `public_state`

字段来自 `backend/internal/events/hub.go` 的 `publicStatePayload`：

- `totalVotes`
- `leaderboard`
- `roomId`
- `boss`
- `bossLeaderboard`
- `announcementVersion`

说明：

- `leaderboard` 只有 `includeLeaderboard=true` 时才发送
- 常规点击后的 `flushPublic` 走 `includeLeaderboard=false`
- 也就是说点击高频链路里通常不带总榜，但仍会带 `boss`、`bossLeaderboard`、`announcementVersion`

#### `room_state`

字段来自 `backend/internal/core/room.go` 的 `RoomList`：

- `currentRoomId`
- `switchCooldownRemainingSeconds`
- `rooms`

其中 `rooms[i]` 当前包含：

- `id`
- `displayName`
- `current`
- `joinable`
- `onlineCount`
- `cycleEnabled`
- `queueId`
- `currentBossId`
- `currentBossName`
- `currentBossStatus`
- `currentBossHp`
- `currentBossMaxHp`
- `currentBossAvgHp`
- `cooldownRemainingSeconds`

#### `online_count`

只包含：

- `count`

### 2.3 大厅和房间内当前分别收到什么

#### 大厅匿名视角

大厅匿名连接当前只会收到：

- `public_state`
- `online_count`

特点：

- `public_state` 使用全局 `snapshot`
- 不会收到按昵称定向的 `room_state`

#### 房间内已登录玩家视角

已登录玩家当前会收到：

- `public_state`
- `room_state`
- `online_count`

特点：

- `public_state` 不是大厅同一份，而是按昵称调用 `GetSnapshotForNickname(...)` 后再发送
- `room_state` 还会额外带整份房间列表

### 2.4 当前一次公共 flush 的实际扇出

当前 `broadcastPublic(...)` 一次执行会产生：

1. 给所有匿名大厅订阅者发送 1 份全局 `public_state`
2. 遍历所有活跃昵称，给每个玩家发送 1 份房间视角 `public_state`
3. 再给每个玩家发送 1 份 `room_state`

也就是说，问题不是“只多发一条公共消息”，而是：

- 高频点击
- 触发一次公共 flush
- 立即放大成多类消息和多目标广播

## 三、当前瓶颈判断

当前带宽压力主要来自三件事叠加：

### 3.1 尾沿防抖导致手感和带宽二选一

当前实现是“重置型尾沿防抖”：

- 连续点击时，旧 timer 被停止
- 新 timer 重新开始计时
- 只有停止点击并安静满 `debounce_ms` 才会真正 flush

后果：

- `200ms` 更省广播，但很容易让公共态长期不刷新
- `100ms` 体感更好，但 flush 次数上升

### 3.2 `room_state` 被绑定到点击公共广播

`room_state` 当前不是一个只在切房、房间在线人数变化时才发的低频消息，而是被放进公共 flush 一起发送。

这会导致普通点击也反复下发：

- 当前房间 ID
- 切房冷却
- 全部房间列表
- 每个房间的 Boss 摘要和在线人数

对点击高频主链路来说，这些字段大多不是强相关。

### 3.3 房间玩家存在“按人重复公共态”

大厅公共态只发一次给匿名订阅者，但已登录玩家的公共态会按昵称逐个生成、逐个发送。

当在线玩家数上升时，广播成本更接近：

`O(匿名订阅者 + 活跃昵称 x 2类消息)`

而不是单纯的 “1 次公共广播”。

## 四、方案目标

本方案追求三个结果同时成立：

1. 点击确认和伤害反馈继续即时
2. 公共态广播频率有硬上限
3. 高频点击链路只发送真正必要的字段

明确不做：

- 不把点击上报改成客户端批量聚合
- 不牺牲 `click_ack` 的即时性
- 不顺手重构整个实时协议

## 五、方案一：公共态从防抖改为节流

### 5.1 目标

把当前“停手才发”的模型改成“窗口内最多发一次”的模型。

建议窗口：

- 首选 `100ms`
- 次选 `120ms` 或 `150ms`

### 5.2 推荐实现

推荐采用“固定窗口尾刷节流”：

- 保留 `publicDirty`
- 新增 `lastPublicFlushAt`
- 收到公共态变更时只标脏
- 如果距离上次 flush 已超过窗口，立即 flush
- 如果还没超过窗口，只安排一个到点补发 timer
- 窗口内再来更多点击，不再不断重置 timer

### 5.3 相比当前尾沿防抖的收益

- 高频点击时广播频率有上限
- 不会因为连续点击把公共刷新无限往后推
- 带宽比 `100ms` 尾沿防抖更可控
- 手感比 `200ms` 尾沿防抖稳定

### 5.4 为什么不继续依赖大防抖

继续把 `debounce_ms` 从 `100` 提到 `150`、`200`、`300`，本质上仍是：

- 用更晚的公共态刷新换更低广播频率

这条路可以省带宽，但无法同时保住体感，因此不适合作为主方案。

## 六、方案二：把 `room_state` 从点击高频广播中拆出

### 6.1 当前问题

普通点击时，`Dispatcher.broadcastPublic(...)` 会顺带给每个活跃玩家发 `room_state`。

但 `room_state` 里的大部分字段只在以下场景真正需要更新：

- 玩家切房成功
- 切房冷却变化
- 房间在线人数变化
- 房间 Boss 轮换或状态切换

### 6.2 建议调整

`room_state` 改成事件驱动或低频驱动：

- 切房接口成功后定向发送
- 登录或首次订阅时发送初始化值
- 房间在线人数变化时发送
- 房间 Boss 生命周期变化时发送
- 普通点击不再触发 `room_state`

### 6.3 预期收益

- 立刻砍掉一次点击对“整份房间列表”的重复广播
- 玩家数量上升时，节省效果明显
- 对手感无负面影响，因为点击本身不依赖房间列表刷新

## 七、方案三：把高频公共态和低频公共态拆层

### 7.1 当前问题

当前 `public_state` 虽然已经把总榜从高频点击链路里拆掉，但仍然混合了：

- 高频变化字段
- 中低频变化字段

例如：

- `boss` 中的 HP、状态属于高频
- `bossLeaderboard` 在高频点击期可能比较大
- `announcementVersion` 明显是低频字段

### 7.2 建议拆分

高频公共态建议保留：

- `totalVotes`
- `roomId`
- `boss` 的最小必要字段
- 必要时只保留 `bossLeaderboardCount`

低频公共态单独通道或单独节奏发送：

- 完整 `leaderboard`
- 完整 `bossLeaderboard`
- `announcementVersion`

### 7.3 建议节奏

- 点击高频广播：`100ms` 节流
- Boss 排行或总榜：`500ms ~ 2000ms` 单独节流
- 公告版本：仅变更时发送

### 7.4 收益

- 高频广播包体更小
- 榜单越大，收益越明显
- 小带宽服务器更容易稳定

## 八、方案四：区分大厅公共态和房间内公共态职责

### 8.1 当前问题

大厅匿名用户和房间内玩家都使用 `public_state`，但两者的关注重点并不完全相同：

- 大厅更偏全局看板
- 房间玩家更偏当前房间战斗态

### 8.2 建议方向

大厅公共态保留：

- 全局 `totalVotes`
- 大厅需要展示的全局 Boss 摘要
- 低频全局榜单

房间内公共态优先保留：

- 当前房间 `roomId`
- 当前房间 Boss 实时态
- 当前房间必要的战斗排行摘要

不建议首轮就大改协议，但应在数据职责上先明确：

- “大厅全局概览” 与 “房间战斗实时态” 不是完全同一类载荷

## 九、推荐实施顺序

### 第一阶段：最低风险止血

1. 把公共态从尾沿防抖改成固定窗口节流
2. 普通点击不再顺带发送 `room_state`

这一阶段就能同时改善：

- `200ms` 卡顿问题
- 高频点击时的重复房间列表广播

### 第二阶段：缩小高频包体

1. 把 `bossLeaderboard` 从点击主链路拆出
2. `announcementVersion` 改成只在变更时发送
3. 评估是否只保留 `bossLeaderboardCount`

### 第三阶段：按视角拆职责

1. 收紧大厅和房间玩家的公共态边界
2. 必要时拆成不同公共消息模型

## 十、验收与观测指标

至少要验证：

- 高频点击时 `click_ack` 延迟不变
- 高频点击时公共态刷新不再“憋住不发”
- 单位时间 `public_state` 次数明显下降并稳定在上限内
- 单位时间 `room_state` 次数明显下降
- 单次高频广播平均字节数下降
- 在线玩家增加时，带宽曲线增长更平缓

建议重点观测：

- 每秒 `public_state` 下发次数
- 每秒 `room_state` 下发次数
- 平均 `public_state` payload 大小
- 平均 `room_state` payload 大小
- `click_ack` p50 / p95 延迟
- 出站总字节数

## 十一、最终建议

本问题不建议继续靠把 `debounce_ms` 调大来解决。

当前最合理的路线是：

1. 用节流替代尾沿防抖，给公共态广播加硬上限
2. 把 `room_state` 从普通点击广播中拆出去
3. 把榜单和公告从高频公共态里继续剥离

这条路线的优点是：

- 改动边界清晰
- 与现有 `click_ack` 主链路兼容
- 能同时兼顾手感和小带宽服务器成本
