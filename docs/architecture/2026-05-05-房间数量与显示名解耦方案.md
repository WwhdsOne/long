# 房间数量与显示名解耦方案

> 状态：已审核，待实现
> 日期：2026-05-05
> 结论：采用最小改造方案。真实房间 ID 固定为连续数字字符串，由配置中的房间数量自动生成；房间显示名独立存 Redis，通过后台管理页维护；旧用户若持有非法房间值，统一回落默认房间。

## 一、背景与问题

当前房间模型把“房间身份”和“房间展示文案”混在一起：

- 配置文件直接写 `room.ids`
- 前端直接把 `room.id` 当显示名展示
- Redis 里的 Boss 当前态、循环状态、玩家当前房间，都把 `room.id` 当真实主键

这会带来一个高风险问题：

- 一旦把房间“改名”理解成修改 `room.id`
- Redis 房间键、Boss 循环状态、玩家当前房间值都会立即失配
- 最终表现为房间状态丢失、房间统一不可加入、旧用户被送回默认房间

本次改造的目标，是把“真实房间主键”和“显示名”彻底拆开，后续不再允许通过改名误伤房间主键。

## 二、目标与非目标

### 2.1 目标

- 配置文件不再手写 `ids: ["1", "2", "3"]`
- 配置文件只写房间数量 `room.count`
- 服务启动时自动生成真实房间 ID：`1..N`
- 房间显示名独立存 Redis，不再复用真实房间 ID
- 后台提供房间显示名编辑入口
- 前台和后台统一优先显示 `displayName`，没有则回退 `房间 N`
- 旧用户若持有非法房间值，不做精确迁移，统一回落默认房间

### 2.2 不做

- 不允许后台或配置修改真实 `room.id`
- 不做旧房间 ID 到新房间 ID 的自动映射迁移
- 不做房间排序、自定义隐藏、额外房间说明等扩展字段
- 不把房间显示名迁到 Mongo 或 Consul

## 三、配置模型调整

### 3.1 新配置模型

房间配置调整为：

```yaml
room:
  enabled: true
  count: 3
  default_room: "1"
  switch_cooldown_seconds: 300
```

约束：

- `count` 必须大于 0
- 启动时自动生成房间 ID：`["1", "2", ..., strconv.Itoa(count)]`
- `default_room` 必须落在 `1..count` 范围内
- 如果 `default_room` 非法，回退为 `"1"`

### 3.2 兼容策略

首版改造后，配置文件中不再要求手写 `room.ids`。

实现上建议：

- 配置结构删除 `IDs []string`
- 增加 `Count int`
- `normalizeRoomConfig` 改为基于 `count` 自动生成连续房间 ID
- `configuredRoomIDs()` 继续返回稳定的 `[]string{"1", "2", ...}`

这样后续任何显示名调整，都不会再触碰真实房间主键。

## 四、真实房间 ID 规则

真实房间 ID 是系统内的稳定主键，不再承载展示语义。

规则如下：

- ID 永远是连续数字字符串：`"1"`、`"2"`、`"3"` ...
- Redis 键、玩家当前房间、Boss 当前态、Boss 循环态，全部继续使用这个稳定 ID
- 前端切房请求继续传真实 `roomId`
- 后台房间管理只允许编辑 `displayName`，不允许编辑 `roomId`

直接影响的现有键包括：

- `player:room:{nickname}`
- `player:room:cd:{nickname}`
- `boss:current:{roomID}`
- `boss:cycle:{roomID}`

因此，后续“改房间名”只会改显示层，不会再触发这些键失配。

## 五、显示名存储模型

### 5.1 Redis 存储

房间显示名单独存 Redis。

建议使用一份哈希：

```text
{ns}room:names
  1 -> 新手线
  2 -> 高压线
  3 -> 摸鱼线
```

约束：

- field 使用真实 `roomId`
- value 为显示名字符串
- 未设置显示名时，不写该 field

### 5.2 默认回退

如果某个房间没有自定义显示名：

- 前端展示 `房间 1`
- 前端展示 `房间 2`
- 前端展示 `房间 3`

也就是说，显示名是可选覆盖层，不是必填主数据。

## 六、接口改造

### 6.1 `GET /api/rooms`

在现有房间列表返回结构中，为每个房间新增：

- `displayName`

规则：

- 如果 Redis 里存在该房间显示名，返回该值
- 如果不存在，后端直接返回默认文案 `房间 N`

推荐口径：

- 后端直接返回 `displayName`
- 若未命名，则返回 `房间 N`

这样前端逻辑最简单，不必自己再拼接回退文案。

### 6.2 后台房间管理接口

新增后台接口：

- `GET /api/admin/rooms`
  - 返回房间列表
  - 字段至少包含：`id`、`displayName`
  - 可附带当前 Boss 名称、循环开关，便于管理页展示

- `PUT /api/admin/rooms/:roomId`
  - 只允许更新 `displayName`
  - 不允许修改 `roomId`

请求示例：

```json
{
  "displayName": "高压线"
}
```

校验约束：

- `roomId` 必须在 `1..count` 范围内
- `displayName` 去首尾空格后可为空
- 空字符串表示清空自定义名，回退默认显示

### 6.3 非目标接口

本次不新增：

- 改房间真实 ID 的接口
- 改房间排序的接口
- 改房间数量的后台接口

房间数量仍由配置文件控制。

## 七、后台页面改造

在现有后台中新增“房间管理”区域。

首版建议最小化：

- 每个房间一行
- 展示：
  - 房间 ID
  - 当前显示名
  - 输入框
  - 保存按钮
- 可以附带展示：
  - 当前 Boss
  - 循环是否开启

交互约束：

- 管理员只能改显示名
- 清空后保存，表示回退默认文案 `房间 N`
- 不在此页面暴露真实 `roomId` 编辑能力

## 八、旧用户处理

本次方案不做旧房间归属精确迁移。

### 8.1 合法旧用户

如果旧用户当前房间值本身就是合法连续 ID，例如：

- `1`
- `2`
- `3`

且仍在新的 `1..count` 范围内，则：

- 保持原房间不变
- 不会被强制踢回默认房间

### 8.2 非法旧用户

如果旧用户当前房间值是非法值，例如：

- 旧的自定义房间主键
- 超出 `count` 范围的房间号
- 空值或脏值

则：

- 服务端读取时自动回落到 `default_room`
- 不做旧值到新值的映射恢复

用户侧表现：

- 下次进入页面后，会被系统送回默认房间
- 之后房间状态恢复稳定，不再继续受旧脏数据影响

## 九、运行态影响

本次改造完成后：

- 房间 Boss 当前态仍按真实 `roomId` 隔离
- 房间循环开关仍按真实 `roomId` 隔离
- 玩家切房冷却仍按真实 `roomId` 工作
- 改显示名不会影响房间 `joinable` 判定

这意味着：

- “房间名”只是展示层标签
- “房间是否可进入”仍取决于该真实房间下的 Boss/循环运行态

如果线上已经因为错误改动真实房间 ID 导致状态错乱，上线本方案后仍需做一次运行态恢复：

- 按新的 `1..count` 房间范围，重新确认每个房间循环状态
- 必要时重新开启循环或手工激活 Boss

## 十、测试策略

### 10.1 后端测试

需要补充：

- `room.count=3` 时自动生成 `1/2/3`
- `default_room` 非法时回退到 `1`
- 玩家当前房间值非法时，会自动回落默认房间
- `GET /api/rooms` 返回正确的 `displayName`
- 房间 `joinable`、循环状态与 Boss 运行态在引入 `displayName` 后保持原语义不变
- `PUT /api/admin/rooms/:roomId` 只能修改显示名，不能修改真实房间 ID
- 清空 `displayName` 后，回退为默认文案

### 10.2 前端测试

需要补充：

- 房间卡片有 `displayName` 时优先显示自定义名
- 没有 `displayName` 时显示 `房间 N`
- 后台房间管理页能加载、编辑、保存房间显示名

### 10.3 手工验证

至少验证：

- 配置只写 `room.count` 时服务正常启动
- 房间列表显示 `1..count`
- 后台改名后前台显示即时更新
- 清空名字后恢复 `房间 N`
- 旧用户带非法房间值进入时回落默认房间
- 切房、Boss 循环、房间可加入判定不受显示名影响

## 十一、实施建议

建议实施顺序：

1. 调整后端配置模型，去掉 `room.ids`，改为 `room.count`
2. 固化真实房间 ID 生成逻辑
3. 增加 Redis 房间显示名读写
4. 扩展 `GET /api/rooms` 与后台房间管理接口
5. 补后台房间管理 UI
6. 补测试
7. 上线后检查各真实房间 `1..count` 的循环/Boss 运行态

## 十二、结论

本方案采用最小改造路径，核心原则只有两条：

- 真实房间 ID 是系统主键，只由 `room.count` 自动生成，不允许人工命名
- 房间显示名是独立的展示层数据，存 Redis，通过后台维护

这样可以同时解决两类问题：

- 运维层面不再需要手写 `ids: ["1", "2", "3"]`
- 业务层面不再会因为“改名其实改了主键”导致所有房间不可进入
