# x1sv 服务迁移总结

**日期：** 2026-05-11
**源服务器：** 阿里云 ECS（47.93.83.136，2核突发实例，1.6G/40G）
**目标服务器：** x1sv（m-sc-kxx.zicp.fun，Xeon D-1581 6C6G 40G SSD）
**迁移发起人：** wwdhs

---

## 一、迁移背景

阿里云 ECS 突发性能实例在 WebSocket 长连接+高并发场景下暴露严重瓶颈：
- CPU 100%积分耗尽后强制限流至基准 10%
- 3.6G RAM 被 Redis(1.5G) + MongoDB + long 服务基本占满
- 磁盘 IO 受限于云盘 bursting 机制

同学免费提供了一台物理机（Xeon D-1581，真·6核独占），决定全量迁移。

---

## 二、迁移清单

| 组件 | 迁移方式 | 源 → 目标 | 数据量 |
|------|---------|-----------|--------|
| Redis | `dump.rdb` + SCP | 阿里云 → x1sv | 370M（清理备份后） |
| MongoDB | `mongodump` → `mongorestore` | 阿里云 → x1sv | 464M |
| Consul | KV export/import | 阿里云 → x1sv | 14M（压缩后） |
| long 服务 | Docker 镜像推送+拉取 | 本地构建 → x1sv | ~300M |
| beszel-agent | 新建注册 | x1sv → beszel hub | 无（新节点） |
| ali-ddns | docker-compose 部署 | 新建 | 无 |
| Let's Encrypt 证书 | acme.sh 签发 | 新建 | 无 |

## 三、关键问题 & 解决方案

### 3.1 初始 SSH 不通
**现象：** 端口 60533 Connection refused
**原因：** 新系统未安装/启动 sshd
**解决：** 通过 Cockpit Web 控制台（zicp.fun:60531）安装 openssh-server 并启动

### 3.2 Docker 无法启动
**现象：** `systemctl start docker` 卡死
**原因：** 需安装 docker-compose-plugin，且需手动 enable docker socket
**解决：** 安装插件，确认 dockerd 启动成功

### 3.3 Redis 数据丢失
**现象：** 恢复后只有 DB0 有数据，原阿里云上 DB0/DB2 的数据丢失
**原因：** Docker 端口映射配置错误，SCP 传输完成后未覆盖正确的 dump.rdb
**解决：** 重新停止容器 → 覆盖 dump.rdb → 启动，所有 key 恢复

### 3.4 Redis 备份文件过大
**现象：** 首次 SCP 传输 2.6G（含 191 个 `dump-*.rdb` 历史备份），占满磁盘
**解决：** 中断传输，在阿里云清理备份文件为仅保留最新，重新打包后 370M

### 3.5 Consul 配置迁移
**现象：** long 服务连接到 x1sv 的 consul 后读不到配置
**原因：** 用户之前在阿里云上通过 Web UI 手动配置的 KV，未包含在 `consul kv export` 中
**解决：** 用户手动通过 consul Web UI 重新输入 `long-produce.config`

### 3.6 long 容器网络互通
**现象：** long 容器在 `docker-compose_app-net`，consul 在 `app-net`，无法互相访问
**解决：** `docker network connect docker-compose_app-net my-consul`，将 consul 接入 long 的网络

### 3.7 Docker 镜像拉取失败
**现象：** `docker.m.daocloud.io`（403 Forbidden），`docker.nju.edu.cn`（timeout）
**解决：** 本地 build 推送到腾讯云 Registry（:5000），再从 x1sv 拉取

### 3.8 beszel agent 被 hub 拒绝
**现象：** x1sv 的 agent 连接 hub 后 KEY/TOKEN 冲突，hub 拒绝
**原因：** 复用了阿里云 ECS 上相同的 TOKEN
**解决：** 在 hub 数据库中为 x1sv 注册新的 KEY（20a9c7-0fd6b128-1859a0-7fd5d72f），指纹自动填充。hub 上最终显示三台机器

### 3.9 GitHub Actions 无法 SCP
**现象：** GitHub Actions workflow 中 `drone-scp` 报 403
**原因：** GitHub Runner IP 被阿里云 fail2ban 封锁，且 Azure DNS 无法解析 zicp.fun 域名
**解决：** fail2ban 解封 IP 范围，DNS 改用 x1sv.wclick.top

### 3.10 系统时区 UTC
**现象：** 容器内全部 UTC+0，日志时间混乱
**解决：** 主机 `timedatectl set-timezone Asia/Shanghai`，重建 Redis 容器添加 `TZ=Asia/Shanghai`。MongoDB 已有 `TZ=Asia/Shanghai` 无需修改

## 四、性能对比

| 指标 | x1sv（Xeon D-1581） | 阿里云 ECS（突发实例） | 倍数 |
|------|:-------------------:|:--------------------:|:----:|
| CPU 核心 | 6 核独占 | 2 核积分制 | 独占 vs 超售 |
| L3 缓存 | 24MB | 共享（约 1-2MB/核） | — |
| RAM | 5.8G | 1.6G | ×3.6 |
| 磁盘 | SATA SSD 裸盘 | 云盘 bursting | — |
| **吞吐量（wrk 基准）** | **533 req/s** | **48 req/s** | **×11.1** |
| 并发测试 | 100 并发 × 20,000 总请求 | 5 并发 × 1,000 总请求 | ×20 压力下仍快 11 倍 |

> 结论：x1sv 在 20 倍并发压力下仍以 11.1 倍吞吐量碾压阿里云，WebSocket 长连接场景下物理机优势彻底体现。
