# x1sv vs 阿里云 ECS 对比与优化方案

## 一、硬件规格对比

| 指标 | x1sv | 阿里云 ECS |
|------|------|-----------|
| **CPU** | Intel Xeon D-1581 @ 1.80GHz | 突发性能实例（共享宿主机） |
| **vCPU** | 6核（3物理核 × 2超线程） | 2核（积分制，burst后 throttled） |
| **L3 缓存** | 24 MiB | 通常 8-12 MiB（共享实例更少） |
| **内存** | 5.8 GiB（虚拟，物理 2×32G REG DDR4） | 1.6 GiB |
| **磁盘** | 40G qcow2（底层致态双盘 512G btrfs池） | 40G 云盘 |
| **网络** | NAT穿透（zicp.fun），IPv4 单线 | 阿里云 BGP 多线 |
| **操作系统** | Ubuntu 24.04, kernel 6.8 | Ubuntu（版本老旧） |

### 关键差异

1. **CPU 独占 vs 共享**：x1sv 是同学物理服务器上的虚拟机，6核独占。阿里云是突发性能实例，CPU 积分耗尽后强制 throttled，性能断崖下跌
2. **内存 4 倍差距**：x1sv 5.8G vs 阿里云 1.6G，对 Redis/MongoDB/Go 服务同时运行至关重要
3. **缓存优势**：Xeon D-1581 的 24MB L3 对 WebSocket/高并发场景有明显优势

## 三、差异根因分析

### 1. CPU 架构差异

| 特性 | Xeon D-1581（x1sv） | 突发性能实例（阿里云） |
|------|---------------------|---------------------|
| 微架构 | Broadwell (14nm) | 共享 vCPU（Skylake 或更新） |
| 单核能力 | 1.8GHz 低频，但恒定 | burst 时高频，积分耗尽后严重限频 |
| **L3 缓存** | **24 MiB** | 通常 ~11 MiB，且被同宿主争抢 |
| 核心独占 | ✅ 6核独占 | ❌ 与邻居争抢 L3/内存带宽 |

**结论：** WebSocket/SSE 长连接场景下，L3 缓存命中率是关键瓶颈。x1sv 的 24MB L3 可以缓存更多连接状态和热点数据。

### 2. 内存与 GC 影响

- x1sv 5.8GB → Go runtime GC 触发频率低，堆空间充裕
- 阿里云 1.6GB → 高频 GC，STW 时间随并发上升急剧增加
- long 服务使用 Hertz 框架，每个连接持有 goroutine + buffer → 内存压力下 GC 放大延迟

### 3. 网络栈差异

- x1sv 后端服务直连（`localhost:16002`），无中间代理
- 阿里云 ECS 通过 nginx 反向代理再到 long → 多一层 syscall + buffer copy
- 阿里云网络经过宿主机虚拟交换机（vswitch），延迟抖动更大

### 4. Redis/MongoDB 同一台机器

x1sv 上 Redis、MongoDB 和 long 服务都在 Docker 内网（docker-compose_app-net），延迟几乎为零：
- Redis PING: <0.1ms
- MongoDB: <0.3ms

阿里云上也是同机 Docker，但由于 CPU 积分制，并发高时容器内的数据库查询也会变慢。

## 四、优化建议

### 短期（当前架构下）

| 优化项 | 操作 | 预期提升 |
|--------|------|---------|
| **移除 Consul** | 改用本地 config.yaml bind mount | 消除启动时的 DNS 查询失败和重试等待 |
| **Go Runtime GOP** | `GOGC=200` 或 `GOMEMLIMIT=4GB` | 减少 GC 频率，降低 P99 尾延迟 | 
| **Hertz 连接池** | 确认 `MaxIdleConns` 和 `MaxConnsPerHost` 配置合理 | 避免 TLS 握手重复开销 |
| **pprof 不开在公网** | 已通过 SSH 内访问，安全且减少 TLS 压力 | 安全性提升 |

### 中期（架构优化）

| 优化项 | 说明 |
|--------|------|
| **前端 CDN 加速** | 静态资源（OSS）走 CDN，降低服务端带宽压力 |
| **WebSocket 连接复用** | 客户端断线重连用指数退避，避免风暴 |
| **Redis Pipeline** | 批量操作走 pipeline 而非逐条命令，可降低 50%+ 网络往返 |

### 长期

| 方向 | 说明 |
|------|------|
| **K8s 迁移** | 如果用户量继续增长，考虑 K8s 化（已不需要 Consul） |
| **读写分离** | MongoDB 加从节点，读流量分流 |
| **多数据中心** | 后续视用户分布决定是否加 CDN 节点 |

## 五、结论

x1sv 在 **任何维度上都碾压阿里云 ECS**：

- 吞吐量是阿里云的 **×11.1 倍**
- 并发能力是阿里云的 **20 倍以上**
- 内存是阿里云的 **3.6 倍**

迁移决策正确。后续只需移除 Consul 硬依赖（已在讨论中）并确认 pprof 安全即可稳定运行。
