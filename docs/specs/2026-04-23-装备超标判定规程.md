# 2026-04-23 装备超标判定规程

## 1. 用途

本文给弱模型使用，只解决一个问题：判断一件装备设计是否超标。

固定输出三档：

- `合格`
- `临界`
- `超标`

除这三档外，不输出第四种结论。

## 2. 输入字段

每次只判断一件装备，输入字段固定为：

```json
{
  "slot": "weapon",
  "rarity": "史诗",
  "bonusClicks": 14,
  "bonusCriticalCount": 7,
  "bonusCriticalChancePercent": 1.8,
  "enhanceCap": 6
}
```

字段要求：

- `slot` 只能是 `weapon`、`armor`、`accessory`
- `rarity` 只能是 `普通`、`优秀`、`稀有`、`史诗`、`传说`、`至臻`
- 其余字段都必须是大于等于 `0` 的数字
- `enhanceCap` 必须按整数处理

若字段不合法，直接输出 `超标`，原因写 `输入字段非法`。

## 3. 禁止自由发挥

判断时只允许使用本文规则：

- 不自行发明新稀有度
- 不自行发明新评分体系
- 不因为“整体感觉合理”覆盖硬规则
- 不参考掉率、活动周期、UI 动效、名称包装

若硬规则和主观感觉冲突，以硬规则为准。

## 5. 固定计算步骤

### 5.1 计算基础总量

先算：

- `baseTotal = bonusClicks + bonusCriticalCount + bonusCriticalChancePercent`

### 5.2 计算成长基数

- `growthBase = ceil(baseTotal / 4)`

### 5.3 计算满强化总量估算

按固定近似法：

- `estimatedPerEnhance = growthBase * 0.4 + 0.1`
- `estimatedFullTotal = baseTotal + enhanceCap * estimatedPerEnhance`

保留一位小数即可。

注意：

- 不做随机模拟
- 不逐次回代成长
- 不改公式

## 6. 固定判定步骤

严格按顺序判断。

### 第一步：字段合法性

如果字段缺失、枚举值不合法、出现负数，直接输出：

- `verdict = 超标`
- `reason = 输入字段非法`

### 第二步：硬上限检查

如果出现任一情况，直接输出 `超标`：

- `bonusClicks > clicks_max`
- `bonusCriticalCount > crit_max`
- `bonusCriticalChancePercent > crit_chance_max`
- `enhanceCap > cap_max`

### 第三步：双高项检查

先算每项是否落在本档上半区间。

上半区间定义：

- `value >= (min + max) / 2`

若满足以下条件，输出 `超标`：

- 同一装备有两项基础属性都在本档上半区间
- 且 `enhanceCap` 也在本档上半区间

若满足以下条件，先记为 `临界候选`：

- 同一装备有两项基础属性都在本档上半区间
- 但 `enhanceCap` 还没到本档上半区间

### 第四步：满强化越档检查

如果当前是 `普通` 或 `优秀`，用 `estimatedFullTotal` 对照 `next_total_min` 和 `next_total_mid`：

- `estimatedFullTotal >= next_total_mid`，直接判 `超标`
- `estimatedFullTotal >= next_total_min`，记为 `临界候选`

如果当前是 `稀有`、`史诗` 或 `传说`，用 `estimatedFullTotal` 对照 `next_total_mid` 和 `next_total_max`：

- `estimatedFullTotal >= next_total_max`，直接判 `超标`
- `estimatedFullTotal >= next_total_mid`，记为 `临界候选`

如果当前是 `至臻`，跳过这一步，不做跨档比较。

### 第五步：核心属性压力检查

若核心属性在本档上半区间，且 `enhanceCap == cap_max`，记为 `临界候选`。

### 第六步：输出结论

按以下顺序输出：

- 只要任一步已经命中 `超标`，最终就是 `超标`
- 否则，只要存在任意 `临界候选`，最终就是 `临界`
- 否则输出 `合格`

## 7. 输出格式

输出固定包含 4 个字段：

```json
{
  "verdict": "临界",
  "reasons": [
    "点击和暴击都处于本档上半区间",
    "满强化总量估算进入下一档低位"
  ],
  "violations": [],
  "advice": "若要保留当前点击高位，建议下调暴击或强化上限"
}
```

字段要求：

- `verdict` 只能是 `合格`、`临界`、`超标`
- `reasons` 写触发结论的直接原因
- `violations` 只写硬超标项；没有就给空数组
- `advice` 只给 1 句修正建议

## 8. 示例

### 8.1 合格示例：武器

输入：

```json
{
  "slot": "weapon",
  "rarity": "稀有",
  "bonusClicks": 8,
  "bonusCriticalCount": 3,
  "bonusCriticalChancePercent": 0.8,
  "enhanceCap": 4
}
```

输出要点：

- 不超过本档上限
- 没有两项同时高位再叠高上限
- 结论应为 `合格`

### 8.2 超标示例：武器

输入：

```json
{
  "slot": "weapon",
  "rarity": "稀有",
  "bonusClicks": 11,
  "bonusCriticalCount": 5,
  "bonusCriticalChancePercent": 1.5,
  "enhanceCap": 5
}
```

输出要点：

- 三项基础值都贴近上限
- 至少两项在本档上半区间
- `enhanceCap` 也在本档上半区间
- 结论应为 `超标`

### 8.3 合格示例：护甲

输入：

```json
{
  "slot": "armor",
  "rarity": "史诗",
  "bonusClicks": 4,
  "bonusCriticalCount": 12,
  "bonusCriticalChancePercent": 1.0,
  "enhanceCap": 5
}
```

输出要点：

- 暴击是主属性，职责正确
- 只有核心属性偏高，但未叠满所有项
- 结论应为 `合格`

### 8.4 超标示例：护甲

输入：

```json
{
  "slot": "armor",
  "rarity": "史诗",
  "bonusClicks": 7,
  "bonusCriticalCount": 16,
  "bonusCriticalChancePercent": 2,
  "enhanceCap": 6
}
```

输出要点：

- 点击和暴击都在本档上半区间
- 强化上限也顶格
- 结论应为 `超标`

### 8.5 合格示例：饰品

输入：

```json
{
  "slot": "accessory",
  "rarity": "传说",
  "bonusClicks": 3,
  "bonusCriticalCount": 5,
  "bonusCriticalChancePercent": 4.5,
  "enhanceCap": 6
}
```

输出要点：

- 暴击率是核心属性
- 副属性没有同时贴顶
- 结论应为 `合格`

### 8.6 超标示例：饰品

输入：

```json
{
  "slot": "accessory",
  "rarity": "传说",
  "bonusClicks": 6,
  "bonusCriticalCount": 9,
  "bonusCriticalChancePercent": 6.5,
  "enhanceCap": 7
}
```

输出要点：

- 两项以上基础属性高位
- 强化上限顶格
- 结论应为 `超标`

## 9. 一句话判定模板

如果要把本文直接喂给弱模型，可要求它固定按这句话执行：

> 先查表，再算 baseTotal、growthBase、estimatedFullTotal，然后按字段合法性、硬上限、双高项、满强化越档、核心属性压力的顺序判断，只输出合格、临界、超标三档结论和对应原因。
